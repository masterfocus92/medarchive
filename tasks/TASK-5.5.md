### T5.5.2-BE — Контекст под кит v2: вариант бейджа и AI-поля
Блокирует: T5.5.3, T5.5.6
Заблокирован: —

**Цель:** шаблоны получают вариант бейджа и список AI-полей готовыми — логика отображения в одном месте, не в Jinja.

**Scope:**
- `PARSE_STATUS_LABELS[PARSED]` → «разобрано — проверьте» (решение 15.07.2026).
- `app/services/ui.py` — хелперы отображения (паттерн хэндоффа, наша реализация): `badge_for(record) -> (kind, label)` — подтверждена → `("done", "подтверждено")`, иначе (`uploaded`|`parsing` → `"pending"`, `parsed` → `"review"`, `parse_failed` → `"failed"`) с подписью из `PARSE_STATUS_LABELS`; `none` → `(None, None)`. Мэппинг вариантов — рядом с подписями в `models/record.py`.
- `ai_fields(record) -> set[str]`: непустые поля из `_DRAFT_FIELDS` при `parse_status == parsed AND confirmed_at IS NULL`, иначе пусто. Признак точный: до подтверждения поля записи наполняет только конвейер (форма создания пишет лишь заметку и пациента). Без миграций; `raw_response` не используется (ADR-013).
- Контексты: лента — `records[*].status_kind` (None у подтверждённых, бейджа нет); карточка — `status_kind="done"`; экран проверки — `status_kind`, `status_label`, `ai_fields`.

**Явно НЕ входит:** шаблоны и CSS; htmx/staging (не вводятся); колонка `ai_filled` (отклонена — выводим из имеющихся данных).

**Контракт:** лента `records: [{id, title, event_date, created_at, record_type, clinic, status_label|None, status_kind|None, confirmed}]`; проверка: `+ status_kind, ai_fields: set[str]`; карточка: `status_label="подтверждено", status_kind="done"`.

**Критерии приёмки:**
- Тесты `badge_for`: все четыре `parse_status`, подтверждённая, `none`.
- Тесты `ai_fields`: parsed-неподтверждённая → ровно непустые draft-поля; `parse_failed` → пусто; подтверждённая → пусто; `comment`/пациент не входят никогда.
- Подпись «разобрано» как самостоятельный лейбл parsed нигде не осталась (существующие тесты обновлены).
