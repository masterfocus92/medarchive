# TASK-5 — Этап 5: Лента и карточка записи

> Результат этапа: жена у кабинета врача открывает продукт, переключается на Арину, находит в ленте нужную запись (сортируя по внесению или по событию), открывает карточку, пролистывает страницы скроллом и показывает врачу оригинал. Продукт научился отдавать, а не только копить.
>
> Источник истины: **`docs/flows/record-viewing.md`** (утверждён 15.07.2026). Все решения ❓1–8 зафиксированы: двойная сортировка · блок «Ждут проверки» упраздняется (статус-бейджи в ленте) · компактный элемент без сниппета · вертикальный поток страниц · скачивание `<a download>` · PDF без серверного превью · без пагинации · карточка read-only (редактирование/удаление — Э6).

---

### T5.1-BE — Лента: репозиторий и контекст главной
Блокирует: T5.2-FE
Заблокирован: T4.4-BE (принят)

**Цель:** главная отдаёт записи активного профиля в двух сортировках; временные счётчик и блок «Ждут проверки» уходят.

**Scope:**
- `repositories/records.list_by_patient(session, patient_id, sort)`: `sort="created"` → `created_at` desc; `sort="event"` → `COALESCE(event_date, created_at)` desc. **Фильтр soft delete — по умолчанию (инвариант).** Все статусы (неподтверждённые — в ленте с бейджами, ❓2).
- Контекст `GET /` (`?sort=created|event`, невалидное значение → дефолт): `records: [{id, title, event_date, created_at, record_type, clinic, status_label|None, confirmed}]` + `sort`. `records_count` и `awaiting_review` удаляются из контекста и шаблона.
- `status_label` — по правилам ADR-012 (подтверждена → None: бейдж не нужен; иначе русская подпись конвейера).

**Явно НЕ входит:** вёрстка (T5.2), карточка (T5.3/T5.4), фильтры по типу/клинике (Could, бэклог).

**Критерии приёмки:**
- Обе сортировки корректны, включая записи без `event_date` (встают по дате внесения).
- Удалённая запись (проставить `deleted_at` руками) в ленте отсутствует.
- Лента — только активного профиля; переключение профиля меняет состав.
- Тесты на все пункты.

### T5.2-FE — Вёрстка ленты
Блокирует: —
Заблокирован: T5.1-BE

**Цель:** лента читается с телефона одним взглядом: что, когда, чьё состояние.

**Scope:**
- Элемент ленты — компактная композиция `record` кита (первое использование `rec-head`/`rec-meta` в продукте): название serif (или «Запись без названия»), дата mono (событ< или внесение — какая есть), чип типа, клиника, статус-бейдж для неподтверждённых (❓2/3). Весь элемент — ссылка на `/records/{id}`.
- Переключатель сортировки: две тихие ссылки «по внесению | по событию», активная выделена (aria-current).
- Кнопка «Добавить запись» остаётся над лентой; пустое состояние — как есть.
- Убрать из `index.html` счётчик и блок «Ждут проверки» (их CSS — тоже, если не используется больше нигде).

**Примитивы кита:** `record` (компакт: `rec-head`, `rec-date`, `rec-meta`), `chip`, `btn-quiet`. Новые — только через `DESIGN.MD` §8.

**Критерии приёмки:**
- Чек-лист `DESIGN.MD` §9 (стражи, mono-даты, одна primary).
- Тесты: рендер элементов, бейдж только у неподтверждённых, переключатель меняет порядок, пустое состояние живо.

### T5.3-BE — Карточка записи: роут и ветвление
Блокирует: T5.4-FE
Заблокирован: T5.1-BE

**Цель:** `GET /records/{id}` сам решает, что показать: подтверждённой записи — карточку просмотра, неподтверждённой — экран проверки (t₂ потока).

**Scope:**
- Ветвление в существующем роуте: `confirmed_at IS NOT NULL` → шаблон `records/view.html`; иначе — `records/record.html` (экран проверки, как сейчас).
- Контекст карточки: `record` (все поля), `status_label`, `pages: [{position, url, mime, is_pdf, available}]` — `available` по `path.is_file()` (ветка B7: плашка вместо битой картинки + warning-лог), `page_count`.
- Доступ — только своя семья (уже так).

**Явно НЕ входит:** вёрстка (T5.4), кнопки редактирования/удаления (Э6).

**Критерии приёмки:**
- Подтверждённая → карточка; неподтверждённая → экран проверки; чужая/удалённая → 404.
- Файл, отсутствующий на диске, помечен `available=False` (+ лог), страница карточки не 500.
- Тесты на ветвление и контекст.

### T5.4-FE — Вёрстка карточки: страницы вертикальным потоком
Блокирует: —
Заблокирован: T5.3-BE

**Цель:** карточку можно читать и показывать врачу: факты чернилами, пометки карандашом, страницы листаются скроллом.

**Scope:**
- `records/view.html` — полная композиция `record` кита: `rec-head` (название + чип типа + дата события mono), `rec-meta` (клиника, врач, дата внесения), `rec-content` (содержание — чернила), `pencil-note` (заметка — карандаш, только если есть), статус-строка (для подтверждённых — «подтверждено»).
- **Страницы вертикальным потоком (❓4):** фото — `<img>` на ширину колонки, ленивые (`loading="lazy"`), подпись «стр. N из M» mono; тап по фото — оригинал в новой вкладке; PDF — плашка «PDF · открыть» (нативный просмотрщик, ❓6); недоступный файл (B7) — плашка «файл недоступен», не битая картинка.
- «Скачать» — `<a download>` у каждой страницы (❓5).
- Назад в ленту — `btn-quiet`.

**Примитивы кита:** `record` (полная композиция — `rec-content`, `pencil-note` первые использования), `chip`, `btn-quiet`. Стили страниц — расширение блока `.pages` из T3.4.

**Критерии приёмки:**
- Чек-лист `DESIGN.MD` §9; содержание и заметка визуально различимы без подписей (главное правило системы).
- Многостраничная запись скроллится, фото кликабельны, download работает; PDF открывается просмотрщиком.
- Тесты рендера: композиция, чернила/карандаш, страницы, плашки PDF/недоступного файла.

## Порядок работы

```
T5.1-BE ──┬── T5.2-FE
          └── T5.3-BE ── T5.4-FE
```

Этап закрыт, когда: лента активного профиля с двумя сортировками и бейджами; тап по подтверждённой записи открывает карточку, страницы листаются скроллом, оригинал открывается и скачивается; неподтверждённая ведёт на проверку; удалённые невидимы; временные блоки главной (счётчик, «Ждут проверки») исчезли.
