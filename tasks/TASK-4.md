# TASK-4 — Этап 4: AI-разбор

> Результат этапа: сфотографированный документ через минуту сам становится черновиком записи — поля предзаполнены, пациент предложен; человек открывает экран проверки, правит одним тапом и подтверждает. Провал AI не мешает ничему (инвариант). Качество проверяется на **реальных документах семьи** — это риск №2 спеки, этап существует, чтобы его измерить.
>
> Источники истины: `docs/flows/record-creation.md` (t₃–t₆, статусная модель §6), ADR-012, **ADR-013** (доменный интерфейс экстрактора), **ADR-014** (провайдер — RouterAI, OpenAI-совместимый API; модель `anthropic/claude-sonnet-5`; размен приватности принят владельцем).

**Решения потока ❓4–6, фиксируются нарезкой:**
- ❓4 — обновление статуса в UI: **без JS-поллинга** — статус виден при загрузке страницы; на экране записи при активном конвейере (`uploaded`/`parsing`) — `<meta http-equiv="refresh" content="5">`: страница сама переспрашивает сервер, ноль кода. htmx подключим при первой реальной нужде в частичном обновлении, не раньше (ADR-011-стиль).
- ❓5 — фон-исполнитель: **`BackgroundTasks` FastAPI** (тот же процесс). Осознанное ограничение POC: перезапуск сервера теряет задачу — лечится ретраем (❓6), отдельный воркер — когда семья упрётся, не раньше.
- ❓6 — ретрай: **входит в этап** — кнопка «Разобрать ещё раз» на экране записи при `parse_failed` и при зависшем конвейере (`uploaded`/`parsing` дольше 10 минут). Это же — страховка ограничения ❓5.

### T4.1-BE — Контракт экстрактора и журнал прогонов
Блокирует: T4.2-BE, T4.3-BE
Заблокирован: T3.5-BE (принят)

**Цель:** домен получает язык для разговора с любым экстрактором (ADR-013), а каждый прогон разбора оставляет след — датасет качества и отладки.

**Scope:**
- `ExtractionResult` (pydantic DTO): название, дата события, клиника, врач, тип, содержание, `suggested_patient_id` (или None по каждому полю — экстрактор честно молчит о том, чего не разобрал).
- `Extractor` — Protocol: `extract(files: list[(mime, bytes)], family: list[FamilyMember]) -> ExtractionResult`.
- Конфиг (ADR-014): `EXTRACTOR_PROVIDER` (`openai_compatible` | `claude` | `disabled`), `EXTRACTOR_BASE_URL`, `EXTRACTOR_MODEL`, `EXTRACTOR_API_KEY`. При включённом провайдере неполный конфиг роняет старт (громко, как SECRET_KEY); `disabled` — приложение живёт, конвейер честно кладёт `parse_failed` («экстрактор не настроен») — машина без ключа остаётся рабочей. `.env.example` дополнен значениями RouterAI (без ключа).
- **Таблица `extraction_runs`** (миграция): record_id, provider, model, status (ok/error), raw_response (JSONB, **артефакт провайдера — непереносим**, бизнес-логика не опирается), error, started_at/finished_at. Несколько прогонов на запись — история ретраев.
- Колонка `health_records.suggested_patient_id` (nullable FK) — предложение AI отдельно от выбора человека (`patient_id` не затирается молча — ветка B7 потока).

**Критерии приёмки:** миграция up→down→up (тест с данными — урок T3.5); фейковый Extractor реализует Protocol в тестах; старт без ключа при provider=claude — понятная ошибка.

### T4.2-BE — Адаптер OpenAI-совместимого провайдера (RouterAI)
Блокирует: T4.3-BE
Заблокирован: T4.1-BE

**Цель:** первый настоящий экстрактор: русский меддокумент (фото/PDF) → `ExtractionResult` через RouterAI (ADR-014).

**Scope:**
- **Шаг 0 — проверка vision живым запросом** (до всего остального): фото → `anthropic/claude-sonnet-5` через `/chat/completions` с base64 data-URI. Не работает → стоп, разговор с владельцем о смене агрегатора.
- `OpenAICompatibleExtractor`: `httpx` напрямую (openai-SDK не ставим), `Authorization: Bearer`, промпт **внутри адаптера** (ADR-013): извлекать и цитировать, не интерпретировать (граница AI, спека §4); пациента предлагать из списка семьи (ФИО+даты рождения), «не уверен» = None.
- **Структурированный ответ без надежды на прослойку:** инструкция «только JSON» + валидация ответа pydantic'ом + один повтор при невалидном JSON. Если живой запрос покажет поддержку `response_format`/tools — использовать, но fallback обязателен (ADR-014: посредник ничего не гарантирует).
- **Подготовка изображений — `services/imaging.py`** (отдельный сервис, пригодится Э5): HEIC → JPEG (`pillow` + `pillow-heif`, обязательство ❓8), **PDF → PNG постранично** (`pypdfium2` — RouterAI принимает только png/jpeg/webp/gif). Даунскейл до разумной стороны (телефонные 12Мп жмут токены).
- Таймаут и перевод HTTP/сетевых ошибок в доменные (конвейер не знает про httpx).
- Модель и base_url — из конфига; фактическая модель пишется в `extraction_runs.model`.

**Явно НЕ входит:** вызовы из конвейера (T4.3), UI, родной адаптер Anthropic (бэклог — включится сменой конфига). Никаких вызовов API из роутов — инвариант.

**Критерии приёмки:** юнит-тесты на сборку запроса и разбор ответа (фикстуры, без сети), включая невалидный JSON → повтор → доменная ошибка; конвертация HEIC и PDF покрыта тестами; **ручная проверка владельцем на 2–3 реальных документах** через одноразовый скрипт (`uv run python -m app.tools.try_extractor <файлы>`) — качество глазами, до конвейера (риск №2).

### T4.3-BE — Конвейер разбора
Блокирует: T4.4-BE
Заблокирован: T4.2-BE

**Цель:** статусы задвигались сами: `uploaded → parsing → parsed / parse_failed`, черновик полей ложится в запись.

**Scope:**
- Сервис `run_extraction(record_id)`: переходы статусов по ADR-012 (терминальные не затираются), прогон в `extraction_runs`, черновик — в пустые поля записи + `suggested_patient_id`. Поля, уже заполненные человеком, **не перезаписываются**.
- Запуск: `BackgroundTasks` после `create_record` с файлами (❓5). Ошибка любого рода → `parse_failed` + запись ошибки в прогон; **сохранение записи не откатывается никогда** (инвариант).
- Ретрай (❓6): `retry_extraction(record_id)` — разрешён из `parse_failed` и из зависших `uploaded`/`parsing` (>10 мин от `created_at`/начала прогона); новый прогон в журнале.

**Критерии приёмки:** тесты с фейковым Extractor: успех (черновик лёг, parsed), провал (parse_failed, поля пусты, запись жива), таймаут, ретрай из failed и из зависшего, запрет ретрая活ого конвейера; человеческие правки не затёрты повторным разбором.

### T4.4-BE — Экран записи/проверки: роуты и подтверждение
Блокирует: T4.5-FE
Заблокирован: T4.3-BE

**Цель:** запись можно открыть, проверить черновик, поправить и подтвердить — подтверждение по ADR-012.

**Scope:**
- `GET /records/{id}` — экран записи: поля формой (черновик/пусто), статус текстом (предикаты из ADR-012), файлы (ссылки на страницы — листалка будет в Э5), meta-refresh при активном конвейере (❓4). Только своя семья (404 неразличимо).
- `POST /records/{id}/confirm` — сохранить поля + `confirmed_at=now()`; принятие AI-пациента = обычная правка `patient_id`.
- `POST /records/{id}/reparse` — ретрай (условия — T4.3).
- Главная: блок **«Ждут проверки: N»** со ссылками на записи (`parsed`/`parse_failed` без `confirmed_at`) — вход на экран проверки до появления ленты (Э5 заменит).

**Контракт (стык с T4.5-FE):** контекст экрана записи: `record` (все поля), `status_label`, `files: [{position, url}]`, `members` (+ активность), `suggested_patient` (если есть и ≠ выбранному), `can_retry: bool`. POST-формы выше.

**Критерии приёмки:** тесты: свой/чужой доступ; подтверждение ставит confirmed_at и сохраняет правки; confirm повторно — идемпотентно обновляет поля, confirmed_at не сдвигается ❓решить в плане; reparse по правилам; блок «ждут проверки» появляется и исчезает.

### T4.5-FE — Вёрстка экрана проверки
Блокирует: —
Заблокирован: T4.4-BE, T1.4-FE (принят)

**Цель:** проверка черновика — минимум трения: всё предзаполнено, правка в один тап, одна кнопка.

**Scope:**
- Поля на примитивах: `field`, **`control.data` для даты события (моноширинная — первое использование)**, **`chip` для типа записи (первое использование: частые варианты + своё значение)**, textarea-карандаш для комментария.
- Статус разбора — текстом всегда (без спиннеров); при `parse_failed` — текст «что делать» + кнопка «Разобрать ещё раз» (`btn-secondary`).
- AI-пациент: если предложение отличается от выбранного — подсветка «AI считает: Арина» с принятием одним тапом.
- Одна `btn-primary` «Сохранить» → тост «Запись сохранена». Meta-refresh при `uploaded`/`parsing` (страница живёт сама).
- Страницы записи — простым списком ссылок (миниатюры/листание — Э5).

**Примитивы кита:** `field`, `control.data`, `chip`, `btn-primary/secondary/quiet`, `toast`, `pencil-note`/`textarea.pencil`. Новые — только через `DESIGN.MD` §8.

**Критерии приёмки:** чек-лист `DESIGN.MD` §9 (даты — mono!); все статусы видны текстом; принятие AI-пациента — один тап; тесты рендера по контракту + стражи.

## Порядок работы

```
T4.1-BE ── T4.2-BE ── T4.3-BE ── T4.4-BE ── T4.5-FE
```

Цепочка строго последовательная. Этап закрыт, когда: реальный документ семьи → «разобрано» → экран проверки с черновиком → правка → «Сохранить» → подтверждена; провал/зависание разбора лечится кнопкой ретрая; в `extraction_runs` — честная история прогонов (наш датасет качества).
