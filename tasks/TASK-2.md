# TASK-2 — Этап 2: Вход и профили

> Результат этапа: жена открывает продукт на телефоне, входит со своей учёткой (один раз — сессия долгоживущая), видит переключатель профилей на монограммах, выбирает дочь — и видит её (пока пустую) карту. Всё, что не `/login` и не `/health`, без входа недоступно.
>
> Спека: `OVERVIEW.MD` §6 «Вход», «Онбординг семьи». Действующие ADR: 001, 005, 006, 007.

---

### T2.1-BE — Seed: семья, три члена, две учётки
Блокирует: T2.2-BE, T2.4-BE
Заблокирован: T1.3-BE (модели)

**Цель:** одна команда создаёт рабочее семейное пространство — учётки, семью и трёх членов — без какого-либо UI регистрации.

**Scope:**
- Скрипт запускается командой (`uv run python -m app.seed` или аналог), использует модели и session-фабрику этапа 1.
- Создаёт: семью · трёх членов (двое взрослых, дочь) · две учётки для взрослых (`is_admin = true` обеим — по спеке оба взрослых админы).
- Пароли хэшируются (passlib/bcrypt — уже в зависимостях).
- **Идемпотентность:** повторный запуск не создаёт дублей и не падает — сообщает, что данные уже есть.
- **Код скрипта в git, данные — вне (решение владельца, 13.07.2026, уточнено):** сам скрипт не содержит ни одной реальной буквы данных и версионируется как обычный код. Все данные (email'ы, пароли, ФИО, даты рождения, пол) — во внешнем файле в `.gitignore`; в репозиторий коммитится только обезличенный шаблон этого файла (`*.example`).

**Явно НЕ входит:** UI управления семьёй (Won't POC), регистрация и восстановление пароля (не существуют по спеке), сам вход (T2.2).

**Контракт:**
- После успешного выполнения в БД: 1 семья, 3 члена, 2 учётки (обе admin), ребёнок — без учётки.
- Имя файла данных, имя шаблона и команда запуска фиксируются в плане тикета и в `docs/code/README.md` (раздел «Окружение и команды»).

**Критерии приёмки:**
- Свежая БД (`alembic upgrade head`) → seed → структура из контракта; повторный запуск — без дублей и без падения.
- Хэш в БД не является паролем (bcrypt), пароль в открытом виде нигде не сохраняется и не логируется.
- **Ни одна реальная буква данных (email, пароль, ФИО, дата рождения) не попадает ни в один коммит:** в коде скрипта данных нет, файл данных — в `.gitignore`, в шаблоне — только очевидные плейсхолдеры.
- Негативный: seed без файла данных (или с незаполненным шаблоном) падает с понятной ошибкой, ничего не создав.

### T2.2-BE — Аутентификация: вход, долгоживущая сессия, защита всего приложения
Блокирует: T2.3-FE, T2.4-BE, все этапы 3+
Заблокирован: T2.1-BE (нужны учётки), T1.2-BE (каркас)

**Цель:** вход по email+паролю превращается в долгоживущую сессию (вход — одноразовый акт на устройство), а всё приложение по умолчанию закрыто.

**Scope:**
- Механизм сессии: подписанная cookie (httponly, samesite; долгий срок жизни — месяцы). Секрет подписи — из конфига (ADR-001), новая настройка в `Settings`.
- `POST /login` — проверка пары email+пароль по `accounts`, установка сессии, redirect на `/`.
- `GET /login` — рендер шаблона входа (сам шаблон — T2.3-FE).
- Зависимость `get_current_account` для роутов; **защита по умолчанию**: любой роут, кроме `/login`, `/health` и `/static`, без сессии отдаёт redirect на `/login`. Новые роуты будущих этапов должны рождаться защищёнными, а не вспоминать про auth.
- `GET /` переводится под защиту.
- **`POST /logout`** (решение владельца, 13.07.2026): сброс сессии → redirect на `/login`. Кнопка в шапке — T2.5-FE; до неё выход проверяется тестом и curl'ом.

**Явно НЕ входит:** восстановление пароля («спроси Дмитрия»), регистрация, переключатель профилей (T2.4).

**Контракт (стык с T2.3-FE):**
- `GET /login` → `login.html`, контекст: `{"error": str | None, "email": str}` (введённый email сохраняется при ошибке).
- `POST /login`, form-поля `email`, `password`: успех → 303 на `/`; неверная пара → re-render `login.html` с `error` (текст говорит, что делать, без «неверный логин ИЛИ пароль»-ребусов — формулировка в T2.3-FE).
- Роуты, работающие с БД, — `def`, не `async def` (ADR-007).

**Критерии приёмки:**
- Вход с валидной парой → `/` открывается; cookie переживает перезапуск браузера (не session-cookie).
- После `POST /logout` сессии нет: `GET /` снова редиректит на `/login`; logout без сессии не падает (тот же redirect).
- Негативные: неверный пароль → форма с ошибкой, сессии нет; несуществующий email → то же поведение и то же время ответа (без утечки «email существует»); `GET /` без сессии → redirect `/login`; `/health` и `/static/*` доступны без сессии.
- Секрета подписи нет в git; приложение без него не стартует (громкая ошибка конфига).
- Тесты на все негативные сценарии выше.

### T2.3-FE — Экран входа
Блокирует: —
Заблокирован: T2.2-BE (контракт), T1.4-FE (layout)

**Цель:** вход с телефона за десять секунд: два поля, одна кнопка, ошибка объясняет, что делать.

**Scope:**
- `login.html` на `base.html`: заголовок serif, поля email и пароль, одна `btn-primary` («Войти» — имя действия сквозное).
- Ошибка входа — через `field.error`/helper по правилам кита; текст говорит, что делать (например: «Пара email–пароль не подошла. Проверьте раскладку и попробуйте ещё раз»), не извиняется.
- Введённый email сохраняется в поле при ошибке (из контекста контракта).
- Автофокус на email, `autocomplete` для менеджеров паролей, тач-цели ≥ 44px (кит даёт паддингами — не ужимать).

**Примитивы кита:** `field` (label + control + helper/error), `btn-primary` (вероятно `btn-lg`). Новые примитивы не вводятся; макрос `field` добавляется в `macros/ui.html` (первое использование).

**Явно НЕ входит:** «запомнить меня» (сессия и так долгоживущая), ссылки «забыли пароль» (не существует по спеке).

**Критерии приёмки:**
- Чек-лист `DESIGN.MD` §9: только токены (тест-страж), одна primary-кнопка, ошибка говорит что делать, фокус видим.
- На 375px форма удобна одной рукой; поля не зумятся iOS (шрифт контрола ≥ 16px уже в ките — проверить фактически).
- Тест: страница `/login` отдаётся без сессии и содержит форму с нужными полями.

### T2.4-BE — Профили: список членов семьи и активный профиль
Блокирует: T2.5-FE, этап 5 (лента по профилю)
Заблокирован: T2.1-BE (данные), T2.2-BE (сессия)

**Цель:** приложение знает, «на кого я сейчас смотрю»: список членов семьи и выбранный профиль, который переживает перезаход.

**Scope:**
- Репозиторий членов семьи (первое чтение из БД — здесь появляется `app/repositories/`): члены семьи текущего пользователя.
- Активный профиль — в сессии; дефолт при первом входе — сам вошедший (его член семьи).
- `POST /profile/{member_id}` — выбрать активный профиль (валидация: id принадлежит семье текущей учётки, иначе 404) → redirect на `/` (или htmx-фрагмент — решить в плане).
- Контекст `GET /` расширяется данными для переключателя.

**Явно НЕ входит:** лента записей профиля (этап 5 — здесь по-прежнему пустое состояние), редактирование членов семьи (Won't POC).

**Контракт (стык с T2.5-FE):**
- Контекст `/`: `members: [{id, full_name, initials, is_active}]` + `active_member` (для заголовка/пустого состояния).
- `POST /profile/{member_id}` — форма без тела; после успеха `/` показывает выбранный профиль активным.

**Критерии приёмки:**
- После входа активен профиль вошедшего; выбор другого — переживает новый запрос и новый вход (или осознанно фиксируем: живёт с сессией — решить в плане и записать).
- Негативные: выбор `member_id` чужой/несуществующей семьи → 404 без смены профиля; POST без сессии → redirect `/login`.
- Тесты на позитив и оба негативных.

### T2.5-FE — Переключатель профилей на монограммах
Блокирует: —
Заблокирован: T2.4-BE (контракт), T1.4-FE (layout)

**Цель:** три лица семьи всегда на расстоянии одного тапа — переключение профиля из шапки любого экрана.

**Scope:**
- `header` в `base.html` наполняется переключателем: `switcher` из `mono-badge` (инициалы, без фото) + подпись-имя.
- Активный — `aria-selected="true"` (семантика, не только класс — `DESIGN.MD` §7), стиль кита.
- Каждый бейдж — кнопка формы `POST /profile/{id}` (работает без JS; htmx-улучшение — по решению плана T2.4).
- Пустое состояние главной персонализируется активным профилем (например: «У Анны записей пока нет»).
- **Кнопка выхода** в шапке: `btn-quiet`, обособлена от переключателя (выход — не действие того же ряда, что выбор профиля); форма `POST /logout`.

**Примитивы кита:** `mono-badge`, `switcher`, `empty`, `btn-quiet`. Макросы добавляются в `macros/ui.html` (первое использование).

**Явно НЕ входит:** управление составом семьи, фото/аватары (их нет в системе — только монограммы).

**Критерии приёмки:**
- Чек-лист `DESIGN.MD` §9 (токены — страж, семантика активного, фокус видим).
- Инициалы корректны для русских ФИО (две буквы: имя + фамилия — уточнить в плане по реальным данным seed).
- Переключение работает без JavaScript (обычная форма); тач-цель ≥ 44px.
- Тест: `/` содержит три монограммы, активная помечена `aria-selected`.

## Порядок работы

```
T2.1-BE ──┬── T2.2-BE ──┬── T2.3-FE
          │             └── T2.4-BE ── T2.5-FE
```

Критический путь: T2.1 → T2.2 → T2.4 → T2.5. T2.3-FE можно делать параллельно с T2.4-BE.

Этап закрыт, когда: свежая БД + seed + вход женой на телефоне → переключение на профиль дочери → её пустая карта → выход возвращает на `/login`; всё вне `/login`, `/health`, `/static` без сессии редиректит на вход; `git log` этапа не содержит ни одного реального email, пароля или ФИО.
