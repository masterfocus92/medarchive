# TASK-6.5 — Этап 6.5 (промежуточный): релизный контур

> Результат этапа: продукт живёт на `mdkarta.online` (прод) и `stg.mdkarta.online` (стенд); релиз = merge в ветку (`stg` — стенд автоматом, `main` — прод с ручным подтверждением); прод бэкапится ночью в шифрованное облако, стенд каждую ночь пересобирается из бэкапа; стенд помечен плашкой. Прод не чувствует ни жизни стенда, ни синков.
>
> Источники истины: **ADR-015** (контур и ветки) · **ADR-016** (бэкапы и prod2stg) · **ADR-017** (нативный PG). Потока нет — этап инфраструктурный, UI-изменение одно (плашка стенда).
>
> **Как выполняются INFRA-тикеты:** артефакт каждого — скрипты/конфиги/юниты в каталоге `infra/` репозитория + пошаговый runbook; серверные шаги выполняются на VPS владельцем (или совместно в сессии), приёмка — по критериям на живом сервере. Секреты (`.env` контуров, ключ rclone-crypt, SSH-ключ деплоя) в git не попадают ни при каких обстоятельствах.

### T6.5.1-INFRA — База VPS: swap, PostgreSQL, контуры, systemd
Блокирует: T6.5.2, T6.5.4, T6.5.5
Заблокирован: —

**Цель:** на VPS существуют два изолированных, автозапускаемых контура приложения с общей нативной БД.

**Scope:**
- Swap 2 ГБ (ADR-017); установка PostgreSQL 16 + `postgresql-16-pgvector` из PGDG; тюнинг под 1 ГБ (`shared_buffers`, `max_connections`) — конфиг-сниппет в `infra/postgres/`.
- Роли и БД: `medcard_prod`, `medcard_stg` (владельцы — отдельные роли с паролями).
- Каталоги `/opt/medarchive/{prod,stg}` (git checkout соответствующих веток), `/var/lib/medarchive/{prod,stg}/files` (`FILES_DIR`), `.env` каждого контура по `infra/env.prod.example` / `env.stg.example` (свои `SECRET_KEY`, свои ключи RouterAI).
- Юниты `infra/systemd/medarchive-{prod,stg}.service`: uvicorn factory-режимом, 1 воркер, порты 8000/8001, `Restart=on-failure`, enabled.
- `uv` на сервере; миграции и seed прод-семьи (`.env.seed` заполняется владельцем на месте).

**Явно НЕ входит:** Caddy/домены (T6.5.2), деплой-скрипт (T6.5.4), бэкапы (T6.5.5).

**Критерии приёмки:**
- `curl localhost:8000/health` и `:8001/health` — 200 у обоих; сервисы переживают `systemctl reboot` (enabled).
- Прод-семья засеяна, вход работает (проверка curl'ом или через SSH-туннель).
- Существующая PWA владельца работает как раньше; `free -h` показывает swap.
- В git — только `infra/` шаблоны и runbook, никаких секретов.

### T6.5.4-INFRA — deploy.sh
Блокирует: T6.5.7
Заблокирован: T6.5.1

**Цель:** релиз любого контура — одна команда, откат — та же команда со старым коммитом.

**Scope:**
- `infra/deploy.sh <prod|stg> [ref]`: `git fetch` + checkout ветки контура (`main`/`stg`, либо явный ref для отката) → `uv sync --frozen` → `alembic upgrade head` (БД контура) → `systemctl restart medarchive-<контур>` → smoke `curl /health` с ненулевым выходом при провале.
- Идемпотентен; лог с датой/ref в `/var/log/medarchive/deploy.log`.

**Явно НЕ входит:** CI-обвязка (T6.5.7); миграции данных.

**Критерии приёмки:**
- `deploy.sh stg` обновляет только стг-контур (прод-процесс и прод-БД не тронуты — проверяется по pid/логам).
- Провал smoke → ненулевой код выхода и внятное сообщение.
- Откат: `deploy.sh stg <прошлый-коммит>` возвращает стенд на прошлую версию.

### T6.5.5-INFRA — Бэкапы в Yandex Object Storage
Блокирует: T6.5.6
Заблокирован: T6.5.1

**Цель:** прод переживает гибель VPS: ночной шифрованный бэкап БД и файлов в независимое облако, с ретенцией.

**Scope:**
- Бакет в Yandex Object Storage (создаёт владелец, ключи — в конфиг rclone на VPS); rclone-remote + **crypt-обёртка** (ADR-016; ключ crypt — на VPS + оффлайн-копия у владельца).
- `infra/backup.sh`: `pg_dump -Fc medcard_prod` + tar `FILES_DIR` → rclone copy в бакет; ретенция 14 дневных + 6 недельных; ненулевой выход и внятный лог при любой ошибке.
- systemd-timer `medarchive-backup` ~03:00.

**Явно НЕ входит:** бэкап стенда (его данные — производные), бэкап другой PWA владельца.

**Критерии приёмки:**
- После прогона в бакете — свежий дамп и архив файлов, **только в шифрованном виде** (прямое скачивание объекта нечитаемо).
- Тестовое восстановление дампа в временную БД проходит (`pg_restore --list` + фактический restore).
- Ретенция удаляет лишнее и не трогает нужное (проверка на искусственном наборе).
- Таймер активен; `systemctl list-timers` показывает следующий запуск.

### T6.5.6-INFRA — prod2stg из бэкапа
Блокирует: —
Заблокирован: T6.5.5

**Цель:** стенд каждую ночь — свежая копия прода; синк не прикасается к проду; тем самым бэкап ежедневно проверяется на восстановимость.

**Scope:**
- `infra/prod2stg.sh`: скачать последний дамп+архив из бакета → пересоздать `medcard_stg` → `pg_restore` → `alembic upgrade head` кодом стенда (репетиция будущих прод-миграций) → развернуть файлы в stg-`FILES_DIR` → `systemctl restart medarchive-stg`. Громкий отказ, если бэкапа нет или restore упал.
- systemd-timer ~04:00 (после бэкапа) + ручной запуск той же командой в любой момент (требование владельца).

**Явно НЕ входит:** любое чтение прод-БД/прод-файлов (по построению — только бэкап); обратный синк (не существует).

**Критерии приёмки:**
- После прогона стенд показывает вчерашние прод-данные; вход прод-учётками работает.
- Во время синка прод-процесс и прод-БД не имели подключений от синка (проверка по `pg_stat_activity`/логам).
- Запуск при отсутствии бэкапа → ненулевой выход, стенд остаётся в прежнем состоянии (не полуразрушен).
- Данные, внесённые на стенде до синка, штатно исчезают (это норма — плашка T6.5.3 предупреждает).

### T6.5.7-INFRA — GitHub Actions: тесты и деплой по веткам
Блокирует: —
Заблокирован: T6.5.2, T6.5.4

**Цель:** пуш в `stg` сам доезжает до стенда; прод-релиз — merge в `main` + явная кнопка подтверждения; провал тестов блокирует любой деплой.

**Scope:**
- `.github/workflows/tests.yml`: на push/PR во все ветки — ruff (check+format) + полный pytest; Postgres — сервис-контейнер `pgvector/pgvector:pg16` (тот же образ, что в dev, ADR-003).
- `.github/workflows/deploy-stg.yml`: push в `stg` → needs tests → SSH на VPS → `deploy.sh stg`.
- `.github/workflows/deploy-prod.yml`: push в `main` → needs tests → GitHub Environment `production` c required reviewer (владелец) → SSH → `deploy.sh prod`.
- Secrets: `DEPLOY_HOST`, `DEPLOY_SSH_KEY` (отдельный ключ только для деплоя, command-restricted при желании). Ветка `main` — protected (только merge из `stg`).
- Создание ветки `stg` от текущего `main`; фиксация нового процесса работы (feature-ветки) в `RELEASE.md` (T6.5.8).

**Явно НЕ входит:** e2e-тесты в CI (бэклог), деплой по тегам (ветки достаточно — ADR-015).

**Критерии приёмки:**
- Пуш в `stg` с зелёными тестами → стенд обновился без ручных действий; с красными — деплой не запускается.
- Merge `stg` → `main` → job ждёт подтверждения; после Approve прод обновляется.
- Секреты не светятся в логах Actions.

**Критерии приёмки:** владелец проходит релиз по документу без моих подсказок; чек-лист смоука проходится на стенде за ≤10 минут.
