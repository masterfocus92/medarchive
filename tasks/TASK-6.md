# TASK-6 — Этап 6: удаление (soft + каскад) и редактирование

> Результат этапа: ошибку в записи можно поправить (тот же экран проверки в режиме правки), мусорную запись — убрать с подтверждением; удалённое исчезает отовсюду, но физически не стирается и восстановимо руками в БД.
>
> Источник истины: **`docs/flows/record-editing-deletion.md`** (утверждён 15.07.2026). Решения ❓1–7 зафиксированы: страница подтверждения вместо JS-модалки · «Удалить» и на экране проверки · redirect из правки — на карточку · ретрай в режиме правки скрыт · право на действия — у любой учётки семьи · тексты §8.6 · восстановление — Won't.
>
> Каскадная дисциплина почти вся уже живёт (инвариант «фильтр в репозитории по умолчанию», файловый роут отдаёт 404 удалённым с T3.3): этап добавляет только сам акт удаления и входы в него.

### T6.1-BE — Удаление: репозиторий, маршруты, страница подтверждения
Блокирует: T6.3-FE
Заблокирован: —

**Цель:** запись можно мягко удалить с подтверждением; удалённая неотличима от несуществующей.

**Scope:**
- `repositories/records.soft_delete(session, record, account)`: `deleted_at = func.now()`, `deleted_by_account_id = account.id`. Единственная точка удаления.
- `GET /records/{record_id}/delete` — страница подтверждения (минимальный шаблон `records/delete.html`: название записи или «Запись без названия», дата, кнопки-скелет; вёрстка — T6.3). Доступ через `get_for_family` (чужая/удалённая/несуществующая → 404 неразличимо).
- `POST /records/{record_id}/delete` — soft delete → flash «Запись удалена» → 303 на `/`. Повторный POST по уже удалённой → 404 (B3: `get_for_family` фильтрует — двойной сабмит безопасен).
- Работает для подтверждённых и неподтверждённых записей (B2).

**Явно НЕ входит:** вёрстка (T6.3); физическое удаление файлов и строк (Won't — спека §5); восстановление (Won't); чистка эмбеддингов (стык Э7, зафиксирован в потоке §5).

**Контракт:** маршруты выше; контекст `records/delete.html`: `record` (название/даты для подписи) + стандартный `switcher_context`. Тост — «Запись удалена» (flash, контракт T3.2).

**Критерии приёмки:**
- После удаления: записи нет в ленте (обе сортировки); `GET /records/{id}`, `/edit`, `/delete`, файлы — 404.
- `deleted_at` и `deleted_by_account_id` заполнены; физически строка и файлы на месте; `extraction_runs` не тронуты.
- Удаление неподтверждённой — работает так же.
- Чужая запись: `GET/POST /delete` → 404; повторный POST → 404.
- Тесты на все пункты, включая негативные.

### T6.2-BE — Режим правки: маршрут и возврат на карточку
Блокирует: T6.3-FE
Заблокирован: —

**Цель:** подтверждённую запись можно править на существующем экране проверки; после сохранения человек видит результат на карточке.

**Scope:**
- `GET /records/{record_id}/edit`: подтверждённая → `records/record.html` с `edit_mode=True`; неподтверждённая → 303 на `/records/{record_id}` (B5: её «режим правки» — обычная проверка); чужая/удалённая → 404.
- Контекст режима правки: `edit_mode=True`, `can_retry` принудительно `False` (❓4 — ретрай только на первичной проверке), `ai_fields` пуст сам собой (запись подтверждена — гарантия `services/ui`).
- `POST /records/{record_id}/confirm` принимает опциональное поле формы `return_to`: значение `card` → 303 на `/records/{record_id}`; любое иное/отсутствие → 303 на `/` (как сейчас). Белый список — не redirect по произвольному URL.
- `confirmed_at` при пересохранении не переписывается — существующее поведение (ADR-012), фиксируется тестом режима правки.

**Явно НЕ входит:** вёрстка условных блоков шаблона (T6.3); правка файлов/страниц записи (Won't этапа — только поля).

**Контракт:** контекст `records/record.html` += `edit_mode: bool` (дефолт False на обычной проверке); форма confirm může нести `<input type="hidden" name="return_to" value="card">` в режиме правки.

**Критерии приёмки:**
- `/edit` подтверждённой отдаёт экран с предзаполненными фактами, без `field.ai` и без «Разобрать ещё раз».
- «Сохранить» из правки обновляет поля, не трогает `confirmed_at`, ведёт на карточку с тостом «Запись сохранена».
- `/edit` неподтверждённой → 303 на `/records/{id}`; чужой → 404.
- `return_to` с мусорным значением → редирект на `/` (не открытый redirect).
- Тесты на все пункты.

### T6.3-FE — Вёрстка действий: rec-actions, страница-«лист», режим правки
Блокирует: —
Заблокирован: T6.1-BE, T6.2-BE

**Цель:** действия над записью читаются по правилам системы: правка — тихая, удаление — красное, обособленное и всегда с подтверждением.

**Scope:**
- `records/view.html`: `rec-actions` (композиция кита, первое использование) — «Редактировать» `btn-quiet` → `/records/{id}/edit` · «Удалить» `btn-danger` с `margin-left:auto` → `/records/{id}/delete`.
- `records/record.html`: кнопка «Удалить» `btn-danger` — обособленно в конце формы перед `action-bar` (❓2), видна и на проверке, и в режиме правки; в `edit_mode` — ретрай скрыт, «Отмена» ведёт на карточку, hidden `return_to=card`.
- `records/delete.html` — полная вёрстка: «лист» кита (`sheet-modal` как страница): «Удалить запись?» + название/дата записи + «Запись скроется из ленты и поиска.» + «Удалить запись» `btn-danger` (форма POST) и «Отмена» `btn-secondary` → назад (❓1, ❓6).
- CSS: `rec-actions`, `btn-danger` (первые использования — перенос из кита), страничный вариант «листа».

**Примитивы кита:** `rec-actions`, `btn-danger`, `sheet-modal`, `btn-quiet`, `btn-secondary`, `action-bar`, `badge`.

**Критерии приёмки:**
- Чек-лист `DESIGN.MD` §9: удаление красным и обособлено, с подтверждением; одна `btn-primary` на экран (на странице подтверждения primary нет вообще); только токены.
- Тесты рендера: rec-actions на карточке; «Удалить» на проверке и в правке; в режиме правки нет `field.ai` и ретрая, «Отмена» → карточка, hidden `return_to`; страница подтверждения содержит тексты ❓6 и форму POST.
- Ручная проверка: карточка → правка → сохранить → карточка; карточка → удалить → подтвердить → лента с тостом; «Отмена» возвращает без изменений.

## Порядок работы

```
T6.1-BE ──┬── T6.3-FE
T6.2-BE ──┘
```

Этап закрыт, когда: с карточки запись правится (возврат на карточку) и удаляется с подтверждением; с экрана проверки — удаляется; удалённая исчезает из ленты и отдаёт 404 по всем URL, включая файлы; физически ничего не стёрто, `deleted_by` зафиксирован; `confirmed_at` переживает правки; тесты и стражи зелёные.
