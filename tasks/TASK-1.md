### T1.1-INFRA — Окружение: docker, конфиг, тулинг качества
Блокирует: T1.2-BE, T1.3-BE
Заблокирован: —

**Цель:** любой `git clone` + две команды — и у разработчика поднята БД, работает линтер и pre-commit; окружение перестаёт быть устным знанием.

**Scope:**
- `docker-compose.yml`: PostgreSQL 16 с pgvector (образ, в котором расширение уже есть), volume для данных, порт наружу.
- `.env.example` + модуль конфигурации приложения: `DATABASE_URL`, `FILES_DIR`. Реальный `.env` — в `.gitignore`.
- Структура каталогов будущего приложения: `app/` с местами под роуты, сервисы, репозитории, модели, шаблоны, статику (пустые `__init__.py` — достаточно).
- Настройка Ruff в `pyproject.toml` и `pre-commit` с хуком Ruff (lint + format).
- `README.md`: раздел «как поднять окружение» на 5–7 строк.

**Явно НЕ входит:** код FastAPI (T1.2), модели и миграции (T1.3), сиды и учётки (этап 2).

**Контракт:**
- Конфиг читается из окружения одним модулем (`app/config.py` или аналог) — все последующие тикеты берут настройки только оттуда, никаких `os.environ` по коду.
- Имя сервиса БД и креды в compose зафиксированы и совпадают с `.env.example`.

**Критерии приёмки:**
- `docker compose up -d` → в контейнере проходит `CREATE EXTENSION IF NOT EXISTS vector;` без ошибки.
- `uv sync` на чистом клоне отрабатывает; `uv run pre-commit run --all-files` — зелёный.
- `.env` отсутствует в git; `.env.example` достаточен, чтобы собрать рабочий `.env` без вопросов автору.
- Повторный `docker compose up` не теряет данные (volume, не anonymous).

**Подсказки:**
- Стандартный образ `postgres:16` расширения pgvector не содержит — посмотри на официальные образы проекта pgvector.
- Вектор-колонки сейчас не нужны (они в этапе 7) — задача только в том, чтобы расширение было доступно.
- Подумай, чем валидировать конфиг при старте: молчаливый `None` в `DATABASE_URL` всплывёт позже и непонятнее, чем громкая ошибка при запуске.

### T1.2-BE — Каркас FastAPI: приложение, шаблоны, статика, smoke-тест
Блокирует: T1.4-FE, этап 2
Заблокирован: T1.1-INFRA

**Цель:** приложение запускается, отдаёт страницу и healthcheck; появляется место, куда этап 2 добавит auth, а этап 3 — загрузку.

**Scope:**
- Точка входа приложения (заменить заглушечный `main.py`): создание FastAPI-приложения, подключение Jinja2-шаблонов и `/static`.
- `GET /health` → `200 {"status": "ok"}` (без обращения к БД — это liveness, не readiness).
- `GET /` → рендер `templates/index.html` (наполнение — T1.4-FE; здесь достаточно, чтобы шаблон существовал и рендерился).
- Первый pytest: приложение стартует, оба роута отвечают 200 (httpx уже в dev-зависимостях).

**Явно НЕ входит:** подключение к БД (T1.3), auth и сессии (этап 2), содержимое и вёрстка страницы (T1.4-FE).

**Контракт (стык с T1.4-FE):**
- Шаблоны лежат в `app/templates/`, статика — в `app/static/`, доступна по `/static/...`.
- `GET /` рендерит `index.html`, контекст пока пустой; `index.html` наследует `base.html`.

**Критерии приёмки:**
- `uv run uvicorn ...` — приложение стартует и без поднятого docker (БД для этих роутов не нужна).
- `uv run pytest` — зелёный.
- Роуты не содержат бизнес-логики — на этом этапе им просто неоткуда взяться, но структура (роут отдельно от создания приложения) уже задана.

**Подсказки:**
- Реши сразу: фабрика приложения (`create_app()`) или модульный синглтон. Вспомни, что тестам удобно собирать приложение с подменённым конфигом.
- `/health`, дергающий БД, — классическая ловушка: контейнер БД перезапускается, и «мёртвым» становится живое приложение.

### T1.3-BE — Доменные модели и первая миграция
Блокирует: этап 2 (seed, auth), этап 3 (создание записи)
Заблокирован: T1.1-INFRA

**Цель:** доменная модель из `OVERVIEW.md` §5 зафиксирована в схеме БД; миграции воспроизводимы на чистой базе.

**Scope:**
- SQLAlchemy-модели (декларативный стиль 2.0): семья, член семьи, учётная запись, запись о здоровье, файл записи.
- Инициализация Alembic + первая миграция, создающая все таблицы.
- Подключение приложения к БД: engine/session-фабрика на основе конфига из T1.1.
- Поля soft delete у записи (`удалена когда`, `кем`) — сразу в схеме, чтобы этап 6 не потребовал миграции-переделки.

**Явно НЕ входит:** репозитории и запросы (появятся вместе с первым чтением, этап 2–3), seed-данные (этап 2), вектор-колонка эмбеддингов (этап 7), эндпоинты.

**Контракт:**
- Модель «член семьи»: ФИО, дата рождения, пол, признак админа. Наличие учётки — отдельная сущность (email, хэш пароля) со ссылкой на члена семьи, а не boolean-флаг.
- Запись: `автор`, `пациент`, `дата_создания`, опциональные `дата_события`, `название`, `клиника`, `врач`, `тип`, `содержание`, `комментарий`, поля soft delete.
- Файлы записи — отдельная таблица с явным порядком внутри записи (порядок страниц значим).

**Критерии приёмки:**
- На чистой БД: `alembic upgrade head` создаёт схему; `alembic downgrade base` откатывает без остатка; повторный `upgrade` проходит.
- Уникальность email учётки и уникальность порядка файла внутри записи — на уровне constraints, не приложения.
- Инвариант «автор — только член семьи с учёткой» отражён структурно (см. подсказку), а не комментарием в коде.
- `дата_создания` проставляется базой/моделью автоматически, а не передаётся снаружи.

**Подсказки:**
- Куда указывает FK автора — на члена семьи или на учётную запись? Один из вариантов делает инвариант «ребёнок не может быть автором» невыразимым в неверном состоянии. Подумай, какой.
- Инвариант «файл ИЛИ комментарий» на уровне СУБД не выразить (две таблицы) — зафиксируй в коде модели docstring'ом, где он будет обеспечен (сервисный слой, этап 3), чтобы не потерять.
- `тип` — открытое строковое поле по спеке, не enum. Соблазн сделать enum — против спеки.
- Проверь автогенерацию Alembic глазами: она любит терять server_default и индексы.

### T1.4-FE — Layout на токенах: tokens.css, base.html, главная-каркас
Блокирует: все FE-тикеты этапов 2–5
Заблокирован: T1.2-BE (закрыт). ~~Плюс открытый вопрос про `design-kit.html`~~ — кит в `docs/`, вопрос закрыт.

**Цель:** каждый следующий экран верстается внутри готового layout и берёт стиль только из токенов — сырые значения физически не нужны.

**Scope:**
- `tokens.css` в статике — токены, извлечённые из `:root` кита, без изменений значений.
- Подключение шрифтов кита: Source Serif 4, Inter, JetBrains Mono.
- `base.html`: бумажный фон, одноколоночный layout (max-width 780px, центр, mobile-first от 520px), блоки `{% block %}` под заголовок и контент, место под будущий переключатель профилей (пустое).
- `index.html`: пустое состояние на примитиве `empty` — заголовок + одна фраза («записей пока нет» по духу, финальный текст твой).

**Явно НЕ входит:** Jinja2-макросы всех примитивов кита (каждый появится в тикете экрана, который его первым использует), PWA-манифест (этап 8, обкатка), переключатель профилей (этап 2).

**Контракт:** использует стык из T1.2-BE (`base.html` + `index.html`, статика на `/static`). Примитивы кита: `empty`; токены — весь `:root`.

**Критерии приёмки:**
- В `base.html`, `index.html` и CSS экранов нет ни одного сырого значения цвета/размера/шрифта — только `var(--...)` (чек-лист `DESIGN.md` §9).
- На ширине 375px и 1280px колонка ведёт себя по правилам `DESIGN.md` §6.
- `:focus-visible` виден (синие чернила), outline не отключён.
- Пустое состояние — приглашение, не заглушка: заголовок + фраза (кнопку действия добавит этап 3).

**Подсказки:**
- Шрифты: CDN-ссылка на Google Fonts против self-hosted — вспомни, что продукт в итоге живёт на своём VPS и открывается с телефона в клинике с плохой сетью. Реши осознанно, зафиксируй в коммите почему.
- `tokens.css` — копия значений из кита, не творчество: расхождение кита и продакшена — ровно та эрозия, от которой §2 `DESIGN.md` защищает.
