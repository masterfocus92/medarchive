# TASK-3 — Этап 3: Ввод — файл первичен

> Результат этапа: жена у кабинета врача фотографирует справку с телефона → тост «Запись сохранена» → запись со статусом «загружено» и файлами в `FILES_DIR`, доступными только своей семье. Заметка без файла работает тоже. AI-разбора ещё нет (этап 4) — но всё готово его принять.
>
> Источник истины: **`docs/flows/record-creation.md`** (утверждён 14.07.2026) — сценарий до t₂+ε включительно + раздача файлов. Решения ❓1–3, 7–10 потока зафиксированы здесь.

**Решения потока, фиксируются нарезкой:**
- ❓1 — запись без файла минует AI-конвейер: сразу статус `confirmed`.
- ❓2 — после сохранения → главная + тост (карточка появится в Э5).
- ❓3 — пациент по умолчанию = активный профиль; AI (Э4) будет только предлагать.
- ❓7 — типы: jpeg/png/webp/heic/pdf; лимит 20 МБ/файл; хранение `FILES_DIR/{record_id}/{position:02}.{ext}`.
- ❓8 — HEIC принимаем как есть; конвертация — обязательство тикета экстрактора (Э4).
- ❓9 — сжатие на клиенте — в бэклог (Could), не Э3.
- ❓10 — порядок страниц: удалить/добавить заново, без drag.

### T3.1-BE — Статус разбора в модели записи
Блокирует: T3.2-BE
Заблокирован: T1.3-BE (принят)

**Цель:** у записи появляется явный статус конвейера — все пять состояний потока, чтобы этап 4 не потребовал миграции-переделки.

**Scope:**
- Колонка `parse_status` в `health_records`: CHECK по пяти значениям потока — `uploaded`, `parsing`, `parsed`, `parse_failed`, `confirmed` (state-диаграмма в `flows/record-creation.md` §6). `server_default='uploaded'`.
- Миграция (up/down), обновление модели; словарь отображения статусов по-русски («загружено», «разбирается», …) — рядом с моделью или сервисом, одно место.

**Явно НЕ входит:** переходы статусов (Э4), UI статусов (Э4/Э5).

**Контракт:** имена статусов — английские константы (enum/Literal в коде), русские тексты — только представление.

**Критерии приёмки:**
- Цикл up→down→up зелёный; вставка с недопустимым статусом режется CHECK'ом.
- Дефолт `uploaded` проставляется БД без передачи из кода.

### T3.2-BE — Создание записи: сервис, транзакция, роуты
Блокирует: T3.3-BE, T3.4-FE, этап 4
Заблокирован: T3.1-BE, T2.2-BE (принят)

**Цель:** `POST /records` за один запрос атомарно рождает запись: файлы на диске, строки в БД, статус по потоку — «сохранено мгновенно» становится правдой.

**Scope:**
- **`services/records.create_record(db, author_account, patient_id, files, comment)`** — единственная точка создания записи:
  - **инвариант «файл ИЛИ комментарий»** обеспечивается здесь (доменная ошибка при пустом входе) — как обещано docstring'ом модели с T1.3;
  - пациент обязан быть из семьи автора (чужой/несуществующий → доменная ошибка, в роуте 404 — неразличимо, как в T2.4);
  - валидация файлов: тип по ❓7 (белый список, проверять и расширение, и magic bytes/сигнатуру — mime от браузера враньё), размер ≤ 20 МБ, пустые (0 байт) отвергать;
  - **одна транзакция**: запись (flush → id) → файлы на диск `FILES_DIR/{record_id}/{position:02}.{ext}` → строки `record_files` → commit; при любом сбое — rollback **и удаление уже записанных файлов** (компенсация: на диске и в БД — вместе или никак, ветка B6 потока);
  - статус: с файлами → `uploaded`; только заметка → `confirmed` (❓1);
  - `FILES_DIR` создаётся при первом использовании; в `record_files.stored_path` — путь относительно `FILES_DIR` (переносимость, как заложено в T1.3).
- **`GET /records/new`** — рендер шаблона формы (наполнение — T3.4-FE); контекст: `members` + `active_member` (шапка и выбор пациента).
- **`POST /records`** (`def`, multipart) — поля `patient_id`, `comment`, `files` (0..N, порядок = порядок в запросе). Успех → 303 на `/` + флаг тоста в сессии (flash: положил-показал-стёр). Доменные ошибки → re-render формы со введённым (B4, B6).
- **Контекст `GET /`** расширяется: `records_count` активного профиля — после первой записи пустое состояние перестаёт врать (честный интерфейс до появления ленты в Э5).

**Явно НЕ входит:** AI-разбор и переходы статусов (Э4), раздача файлов (T3.3), вёрстка формы (T3.4), лента (Э5).

**Контракт (стык с T3.4-FE):**
- `GET /records/new` → `records/new.html`, контекст `{members, active_member, error: str | None, comment: str}`.
- `POST /records`: form-data `patient_id: int`, `comment: str`, `files: list[UploadFile]`; 303 → `/` при успехе; ошибка → 200 re-render с `error` (текст говорит, что делать) и сохранённым введённым.
- Flash-тост: ключ сессии `flash`, читается и стирается при рендере главной; текст «Запись сохранена» (имя действия сквозное).
- Контекст `/`: + `records_count: int`, `flash: str | None`.

**Критерии приёмки:**
- Фото+заметка, только фото, несколько файлов (порядок сохранён), только заметка (→ `confirmed`) — созданы корректно; `автор` = учётка оператора, `created_at` ставит БД.
- Негативные: пусто → доменная ошибка, в БД и на диске ничего; чужой пациент → 404; запрещённый тип (в т.ч. переименованный `.exe` → `.jpg`) и >20 МБ → ошибка, ничего не создано; сбой на середине записи файлов → в БД нет записи, на диске нет осиротевших файлов.
- Прямой заход `POST /records` без сессии → redirect `/login` (default-deny, само).
- Тесты на все пункты, включая компенсацию файлов (сымитировать сбой).

### T3.3-BE — Раздача файлов: только своей семье
Блокирует: этап 4 (экстрактору нужны файлы), этап 5 (просмотр)
Заблокирован: T3.2-BE

**Цель:** файл записи открывается владельцам семьи и никому больше — инвариант «файлы недоступны без аутентификации, в т.ч. по прямой ссылке» становится проверяемым кодом.

**Scope:**
- `GET /records/{record_id}/files/{position}` (`def`): запись принадлежит семье текущего оператора → `FileResponse` с корректным `Content-Type` и `inline`; иначе **404** (чужая и несуществующая неразличимы — как T2.4).
- Файлы отдаются отсюда и только отсюда — `/static` для них не существует (зафиксировано в ADR-010 и middleware).
- Путь собирается строго из БД (`stored_path` относительно `FILES_DIR`) — никакой сборки из пользовательского ввода (path traversal невозможен по построению).

**Явно НЕ входит:** превью/миниатюры, скачивание архивом, просмотр UI (Э5).

**Контракт:** маршрут выше; 200 с файлом / 404; `Content-Type` из `mime_type` записи.

**Критерии приёмки:**
- Свой файл → 200, верный Content-Type; чужой (вторая семья в тестах) → 404; несуществующие record/position → 404; без сессии → redirect `/login`.
- Запись есть, файла на диске нет (битая синхронизация) → 404 с логом, не 500.
- Тесты на все пункты.

### T3.4-FE — Экран добавления: камера, файлы, заметка
Блокирует: —
Заблокирован: T3.2-BE (контракт), T1.4-FE (принят)

**Цель:** «сфотографировал → запись существует» — весь поток К и Ф из `flows/record-creation.md` §4а работает с телефона одной рукой.

**Scope:**
- Кнопка **«Добавить запись»** (`btn-primary btn-lg`) на главной — в пустом состоянии (примитив `empty` наконец получает своё действие) и при `records_count > 0` (рядом со счётчиком «Записей: N» — временная замена ленты до Э5).
- `records/new.html`: **два входа контента** — «Сфотографировать» (`capture="environment"`, цикл «＋ Ещё страница») и «Выбрать из галереи или файл» (`multiple`, фото+PDF); общий превью-список страниц с удалением (без drag — ❓10); заметка `textarea.pencil`; выбор пациента (монограммы, предвыбран активный — ❓3); одна `btn-primary` «Сохранить»; «Отмена» — `btn-quiet` на главную.
- **Первый JS проекта** — `static/js/record-form.js` (vanilla, без зависимостей): склейка файлов из двух input'ов и циклов камеры в один список (`DataTransfer`), превью, удаление страниц, дизейбл «Сохранить» пока пусто (B4). **Прогрессивная деградация:** без JS остаётся одиночный `input multiple` + серверная валидация B4 с re-render — форма работоспособна.
- Тост «Запись сохранена» на главной (примитив `toast` — первое использование: макрос + CSS из кита).
- Серверная ошибка (B6) — текст из контракта, введённое сохранено.

**Примитивы кита:** `btn-primary btn-lg`, `btn-quiet`, `field`, `textarea.pencil`, `mono-badge`/`switcher`-стиль для выбора пациента, `toast`, `empty`. Новых примитивов нет; если по ходу вёрстки превью-списка страниц понадобится новый — через `DESIGN.MD` §8 (кит → §4 → экран), не по ходу.

**Явно НЕ входит:** сжатие фото на клиенте (❓9 — бэклог), drag-сортировка (❓10), экран проверки и статусы разбора (Э4), лента (Э5).

**Критерии приёмки:**
- Чек-лист `DESIGN.MD` §9 (стражи + одна primary + ошибки говорят что делать + тач-цели ≥ 44px).
- С телефона (или эмуляции DevTools): камера открывается сразу по «Сфотографировать»; «＋ Ещё страница» доснимает; пикер даёт мультивыбор; страницы удаляются; смешивание камеры и файлов работает.
- Без JS: форма отправляется, пустая форма получает серверную ошибку с понятным текстом.
- Тосты и счётчик записей на главной после сохранения; «Записей: N» — моноширинное число (правило дат/чисел кита).
- Тесты: рендер формы (примитивы на месте, страж стилей), наличие обоих input'ов с правильными атрибутами, тост после redirect.
