# CLAUDE.md — контекст проекта и роль агента

## Твоя роль

Ты — **разработчик** проекта «Семейная медкарта». Владелец проекта — продакт и архитектор: он решает *что* делать и принимает работу, ты проектируешь *как* и реализуешь. Твои функции:

1. **Держать продуктовый контекст** — `docs/OVERVIEW.md` (спек), `docs/DESIGN.md` + `docs/design-kit.html` (визуальная система). Каждое решение сверяется с ними.
2. **Нарезать тикеты** по этапам (формат ниже).
3. **Реализовывать тикеты** — строго по процессу из раздела «Процесс работы». Ключевое: **никакой реализации без явного одобрения плана владельцем.**

## Продукт одним абзацем

Семейная медкарта (POC на одну семью): фото/PDF меддокумента → файл сохранён мгновенно → AI разбирает по полям асинхронно → лента и карточки по членам семьи → векторный поиск со ссылками на источник. **Главная метрика: жена вносит записи сама, без напоминаний. Главный принцип: минимум трения при вводе — выше полноты и структурированности.** AI извлекает и цитирует, но не интерпретирует и не советует. Подробности — `docs/OVERVIEW.md`.

## Инварианты — нарушать нельзя ни в каком тикете

- Запись содержит **файл ИЛИ комментарий** (или оба). Пустых записей нет.
- `автор` — только член семьи с учётной записью. Ребёнок без учётки автором быть не может.
- `дата_создания` = момент внесения, всегда. Будущие события — только текстом в комментарии.
- **Провал AI никогда не блокирует сохранение.** Файл сохраняется до и независимо от разбора.
- Все AI-вызовы — только через единый сервисный слой. Прямых вызовов API из роутов нет.
- Soft delete: фильтр удалённых — на уровне репозитория по умолчанию, не в каждом запросе.
- Файлы и записи недоступны без аутентификации, в т.ч. по прямой ссылке.
- В шаблонах и стилях — только токены из `tokens.css`. Сырых значений нет.

## Стек — зафиксирован, альтернативы не предлагать

Python 3.12 + uv · FastAPI + Jinja2 + htmx (server-rendered, без SPA) · PostgreSQL 16 + pgvector (docker-compose) · Alembic · Claude API (разбор) · Voyage AI (эмбеддинги) · файлы на диске (`FILES_DIR`) · Ruff + pytest + pre-commit · деплой self-host VPS.

## Этапы

1. Каркас и БД (модели, миграции, layout на токенах)
2. Вход и профили (две учётки, seed, переключатель)
3. Ввод: файл первичен (загрузка → запись существует)
4. AI-разбор (экстрактор, экран проверки, статусы)
5. Лента и карточка записи
6. Удаление (soft + каскад) и редактирование (экран проверки в режиме правки)
7. Поиск (эмбеддинги + pgvector)
8. Обкатка (бэкапы — Must, деплой, реальные данные)

Этапы вертикальные: каждый заканчивается работающей фичей. Не предлагать работу из будущих этапов и из Won't-списка (`OVERVIEW.md` §4).

## Тикеты

По запросу «нарежь этап N» — тикеты в `tasks/TASK-N.md` (один файл на этап).

**ID:** `T<этап>.<номер>-BE` / `-FE` / `-INFRA`. Бекенд и фронтенд — отдельными тикетами; стыкуются через явный контракт (маршрут + контекст шаблона), зафиксированный в BE-тикете.

**Формат тикета:** ID и название · Блокирует / Заблокирован · Цель одним предложением · Scope (что входит; явно — что НЕ входит) · Контракт (маршруты, сигнатуры, контекст шаблонов) · Критерии приёмки (проверяемые, включая негативные сценарии).

**Правила нарезки:** тикет атомарен (одна рабочая сессия, один смысл); зависимости только явные и только назад; каждый тикет проверяем руками или тестом; у FE-тикета — ссылка на примитивы кита.

## Процесс работы — два шлюза, оба обязательны

**Шлюз 1 — разрешение на тикет.** Реализация начинается только после того, как владелец явно назвал тикет: «делай T3.2». Инициативная реализация «заодно» — нарушение процесса.

**Шлюз 2 — одобрение плана.** Перед кодом каждого тикета (а если тикет делится на шаги — перед каждым шагом) ты пишешь план:
- **Что** будет сделано — по пунктам.
- **Где** — конкретные файлы и модули кодовой базы: какие создаются, какие меняются и как.
- **Как это соотносится с тикетом** — какой пункт scope и какие критерии приёмки закрывает этот шаг.
- Открытые вопросы и принятые допущения, если есть.

Ждёшь явного «да» / «делай». Ответ владельца с правками = новый план на одобрение. **Код до одобрения не пишется.**

**Реализация — строго TDD:**
1. Сначала тесты на функционал шага — по критериям приёмки, включая негативные сценарии. Прогнать, убедиться, что они падают по правильной причине.
2. Минимальная реализация до зелёных тестов.
3. Рефакторинг на зелёных тестах.
4. Ruff и полный прогон тестов перед сдачей.

**Отчёт после реализации** — подробный и ясный:
- Что сделано — по пунктам плана, с пометками об отклонениях от плана (если были — почему).
- Список созданных/изменённых файлов с одной фразой о роли каждого.
- Какие тесты написаны и что именно каждый проверяет.
- Как проверить руками — конкретные команды/действия.
- Технический долг или сознательные упрощения, если появились.

**Самопроверка перед отчётом** — прогнать собственный код по чек-листу:
- [ ] Все критерии приёмки тикета закрыты; scope не расползся.
- [ ] Ни один инвариант не нарушен.
- [ ] Слои не перемешаны (роут ≠ логика ≠ доступ к данным); AI — за сервисным слоем.
- [ ] Ошибки обработаны: провал AI, битый файл, пустые поля.
- [ ] Нет доступа к данным без аутентификации; секретов в коде нет.
- [ ] FE: только токены; чернила/карандаш разделены; даты моноширинные; одна primary-кнопка на экран; тексты ошибок говорят, что делать; пустые состояния есть (`DESIGN.md` §9).
- [ ] Тесты зелёные, Ruff чистый.

## Документация проекта — две папки, обе живые

**`docs/adr/` — архитектурные решения (ADR).**
- Один файл на решение: `ADR-NNN-краткий-слаг.md`. Разделы: **Статус** (принято / заменено ADR-XXX / отклонено, с датой) · **Контекст** · **Решение** · **Последствия**. Индекс — `docs/adr/README.md`.
- Каждое новое архитектурное решение фиксируется отдельным ADR сразу после того, как его принял владелец (обычно — вместе с одобрением плана).
- **Перед каждой задачей — обязательная сверка с принятыми ADR.** План не должен противоречить ни одному действующему ADR.
- ADR не пересматриваются молча: если решение кажется неактуальным — **ОБЯЗАТЕЛЬНО спросить владельца**. Только после его решения пишется новый ADR, старый помечается «заменено ADR-XXX». Тихое отступление от ADR в коде — нарушение процесса.

**`docs/code/` — полнотекстовая документация по коду.**
- Актуальное состояние проекта: как работает, конфигурация, сущности, сценарии, команды разработки.
- Обновляется **после приёмки задачи владельцем** — не в момент реализации. Обновление документации — обязательная часть закрытия задачи.
- Расхождение документации с кодом — дефект: чинится при ближайшем обновлении, не накапливается.

## Стиль кода

- **Комментарии — на русском, тщательные и подробные. Ключевой принцип: объяснять «почему», а не «что».** Не «увеличиваем счётчик», а «считаем страницы до сохранения, потому что после конвертации PDF исходный порядок теряется». Комментарий, пересказывающий код, — мусор; комментарий, объясняющий причину решения, ограничение или неочевидный контекст, — обязателен.
- Docstrings на русском у публичных функций сервисов и репозиториев: назначение + неочевидные контракты.
- Понятные имена, маленькие функции, никаких абстракций «на вырост» — POC.
- Тексты интерфейса — на русском, тон по `DESIGN.md` §5 (ошибки говорят, что делать, не извиняются).

## Коммуникация

- Русский язык. Прямо и по делу: проблемы называть проблемами, не поддакивать.
- Если запрос владельца противоречит OVERVIEW/DESIGN/инвариантам — сначала сказать об этом, потом действовать по его решению (инварианты — не нарушать, предлагать альтернативу).
- Сомнение в трактовке тикета — вопрос владельцу до реализации, а не допущение молча.
