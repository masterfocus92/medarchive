# ADR-011 — JavaScript: vanilla, прогрессивное улучшение, без сборки

## Статус

Принято владельцем 14.07.2026 (нарезка этапа 3, запуск эпика).

## Контекст

До этапа 3 продукт обходился без JS. Экран добавления записи потребовал его впервые: склеить страницы из двух источников (циклы камеры + мультивыбор пикера) в один список для отправки средствами HTML невозможно — `input[type=file]` не накапливает файлы между вызовами.

## Решение

- JS — **vanilla**, без фреймворков, зависимостей и сборки: файлы в `app/static/js/`, подключаются `<script defer>` на экране, которому нужны. htmx из стека подключим, когда появится первый реальный сценарий частичного обновления (статусы разбора, Э4) — не раньше.
- **Прогрессивное улучшение — обязательное свойство:** без JS каждый экран остаётся работоспособным. На форме записи оба входа контента — настоящие `input[name=files]`: камера снимает (один кадр), пикер даёт мультивыбор, сервер валидирует пустую форму. JS добавляет: единый список страниц (`DataTransfer`), превью, удаление, «＋ Ещё страница», дизейбл «Сохранить».
- Скрытие нативных контролов и показ кнопок делает сам скрипт при загрузке — не CSS: нет JS — нет и скрытия.

## Последствия

- (+) Нет сборки, node_modules и версий фреймворков; скрипт читается целиком.
- (+) Плохая сеть/старый браузер деградируют в рабочую форму, а не в кирпич.
- (−) На каждом экране с JS придётся осознанно проектировать «как это работает без него» — это цена, признанная полезной (телефон в клинике).
- Тесты рендера проверяют разметку и подключение скрипта; поведение JS проверяется руками на живом устройстве (браузерной автоматизации в POC нет).
