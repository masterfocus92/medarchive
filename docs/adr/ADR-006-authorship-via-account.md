# ADR-006 — Авторство и админство — свойства учётной записи, не члена семьи

## Статус

Принято владельцем 13.07.2026 (одобрение плана T1.3-BE с правками).

## Контекст

Два инварианта продукта: «автор записи — только член семьи с учётной записью» (ребёнок без учётки автором быть не может) и «без учётки → не админ по определению». При этом член семьи без учётки — норма (дочь), и админ может сам добавлять пациентов в семью.

Если авторство/админство висят на `family_members` (FK автора → член семьи, флаг `is_admin` у члена), оба инварианта превращаются в проверки в коде приложения, которые можно забыть или обойти прямой вставкой.

## Решение

Способность **действовать** отделена от способности **быть пациентом**:

- `family_members` — пациенты; для существования не требуют ничего, кроме семьи. Учётка — строго 0..1 (отдельная таблица `accounts` с unique FK).
- `health_records.author_account_id` — FK на **`accounts`**: запись без учётки-автора невыразима в схеме.
- `is_admin` — колонка **`accounts`**: «админ без учётки» невыразим в схеме.

## Последствия

- (+) Оба инварианта обеспечены структурой БД — нарушение невозможно даже прямым SQL с валидными FK.
- (+) Продуктовое «админ добавляет пациентов» ложится естественно: операция всегда авторизована учёткой с флагом.
- (−) Чтобы показать ФИО автора записи, нужен join через `accounts` → `family_members` (для POC незаметно).
- Если появится роль «няня/родственник» (бэклог) — расширяется `accounts`, модель пациентов не трогается.
