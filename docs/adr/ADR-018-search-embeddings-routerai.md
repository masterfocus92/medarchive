# ADR-018 — Поиск: эмбеддинги через RouterAI, хранение векторов, журнал тюнинга

## Статус

Принято владельцем 19.07.2026 (решения ❓1, ❓7, ❓9, ❓10 потока `flows/search.md`). Изменяет зафиксированный стек: **Voyage AI заменён на RouterAI** для эмбеддингов.

## Контекст

Этап 7 — поиск-чат: вопрос словами → векторный retrieval (pgvector) → генеративный ответ строго по найденным записям с обязательными источниками. Стек предписывал Voyage AI для эмбеддингов, но Voyage не оплатить из региона владельца — ровно та ситуация, из-за которой LLM уже идёт через агрегатор RouterAI (ADR-014); embeddings-модели в каталоге RouterAI есть. Владелец требует максимальной точности поиска и предупредил, что после POC будет тюнить эту часть — архитектура обязана поддерживать эксперименты без переделок.

## Решение

1. **Провайдер эмбеддингов — RouterAI** (OpenAI-совместимый `/embeddings`, тот же base URL, что у экстрактора). Слой — по паттерну ADR-013: доменный Protocol + адаптер `openai_compatible` + фабрика; конфиг `EMBEDDINGS_PROVIDER/BASE_URL/MODEL/API_KEY` (+ `disabled` — поиск честно недоступен, продукт живёт).
2. **Критерии выбора модели** (сверка каталога — в тикете T7.1, выбранная модель дописывается сюда): обязательна качественная работа с **русским** текстом (медзаписи русскоязычные); размерность ≤1536 (объём и скорость pgvector на VPS 1 ГБ); цена за 1M токенов — второстепенна (объёмы семейные). Рабочие кандидаты — класс `text-embedding-3-small` или мультиязычный класс BGE-M3.
3. **Хранение — таблица `record_embeddings`**: `record_id` (PK, FK), `embedding vector(dim)`, `model`, `updated_at`. Отдельная от `health_records`: смена модели/размерности не трогает основную таблицу. Расширение `vector` включается миграцией этапа. Индекс ANN не создаём: на семейных объёмах (сотни строк) точный поиск быстрее и точнее; порог для «не нашёл» и K — конфиг `SEARCH_*`.
4. **Переиндексация — штатный инструмент** `app.tools.reindex`: идемпотентный бэкфилл всех подтверждённых записей; используется при деплое этапа, смене модели, тюнинге.
5. **Генерация ответа — моделью экстрактора** (`anthropic/claude-sonnet-5` через RouterAI, тот же ключ, ❓9): один провайдер, один биллинг; отдельный конфиг не заводится, пока не понадобился.
6. **Журнал `search_queries`** (❓10): вопрос, кандидаты retrieval (id + дистанции), итоговый ответ, момент. Аналог `extraction_runs` — датасет для тюнинга точности; в UI не показывается, не чистится.

## Последствия

- Приватность: текст записей уходит в RouterAI дважды (индексация и генерация) — размен уже принят ADR-014, новых уступок нет.
- Смена модели эмбеддингов = правка конфига + `reindex`; несовпадение `model` в строке с конфигом — признак устаревшего вектора (переиндексировать).
- Векторы и журнал — производные данные: в ночной бэкап попадают вместе с БД автоматически, восстановимы reindex'ом.
- Изменение стека отражается в `CLAUDE.MD` (строка стека) — Voyage из проекта уходит.
