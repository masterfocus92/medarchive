# ADR-013 — AI-экстрактор: доменный интерфейс, провайдер — адаптер

## Статус

Принято владельцем 14.07.2026 (обсуждение перед нарезкой этапа 4).

## Контекст

Инвариант «все AI-вызовы за единым сервисным слоем» защищает от вендор-лока, только если слой — интерфейс домена, а не обёртка над конкретным API. Владелец явно хочет опции на будущее: смена LLM после запуска, self-hosted дистрибутив с любой моделью пользователя (бэклог, OVERVIEW §13). Цена смены должна концентрироваться в промпте и валидации качества (неустранимая часть), а не в коде.

## Решение

1. **`ExtractionResult` — доменный DTO** (pydantic): название, дата события, клиника, врач, тип, содержание, предположение о пациенте. Ни одного термина провайдера за пределами адаптера.
2. **`Extractor` — Protocol**; `ClaudeExtractor` — первый адаптер. Конвейер, статусы, UI знают только Protocol и DTO.
3. **Промпты живут внутри адаптера** — они часть провайдера (тюнятся под модель), не домена.
4. **Сырой ответ модели хранится отдельно** (таблица прогонов разбора) с пометкой: артефакт провайдера для отладки и оценки качества; бизнес-логика на него не опирается; при смене провайдера не мигрирует.
5. **Конфиг-ключ `EXTRACTOR_PROVIDER`** — уже сейчас, с единственным значением `claude`: документация намерения ценой в одну строку.
6. **Родственное обязательство (этап 7):** эмбеддинги хранятся с именем и версией модели — иначе смена эмбеддера превращается в археологию вместо скрипта пере-индексации.

## Явно НЕ делаем (Won't POC, подтверждено владельцем)

Рантайм-переключалка провайдеров, MCP, LiteLLM/прокси-прослойки — инфраструктура под проблему, которой пока нет. Дверь не заперта — этого достаточно.

## Последствия

- (+) Смена LLM = новый адаптер + промпт + валидация качества на реальных документах; данные и система не трогаются.
- (+) Тесты конвейера используют фейковый Extractor — быстрые и без API-ключей.
- (−) Дисциплина: любое «быстренько дернуть Claude» обязано пройти через Protocol, даже когда кажется избыточным.
