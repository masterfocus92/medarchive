# ADR-007 — Синхронный SQLAlchemy (без async)

## Статус

Принято владельцем 13.07.2026 (одобрение плана T1.3-BE).

## Контекст

FastAPI позволяет и async-, и sync-доступ к БД. Async SQLAlchemy требует отдельного драйверного режима, `AsyncSession`, async-фикстур в каждом тесте и внимательности с ленивыми загрузками (`MissingGreenlet`). Нагрузка продукта — одна семья на self-host VPS: единицы запросов в минуту в пике.

## Решение

Синхронный SQLAlchemy: `create_engine` + `sessionmaker` (`app/db.py`), сессия отдаётся FastAPI-зависимостью `get_session()`. Роуты, работающие с БД, объявляются `def` (не `async def`) — FastAPI выполняет их в threadpool, event loop не блокируется. Engine и session-фабрика инициализируются лениво: импорт модуля не требует окружения (согласовано с ADR-001/005).

## Последствия

- (+) Простые тесты и репозитории; ленивые загрузки работают без сюрпризов.
- (+) Один драйверный стек psycopg (sync) — меньше зависимостей.
- (−) Роуты с БД обязаны быть `def`, не `async def` — правило для ревью; `async def`-роут с sync-сессией заблокирует event loop.
- Долгие AI-вызовы (этап 4) к этому решению не привязаны: они за сервисным слоем и могут быть async/фоновыми независимо от доступа к БД.
