# ADR-001 — Конфигурация через pydantic-settings

## Статус

Принято владельцем 13.07.2026 (одобрение плана T1.1-INFRA).

## Контекст

Нужна единственная точка чтения окружения (`DATABASE_URL`, `FILES_DIR`, дальше — ключи AI-провайдеров): разбросанные по коду обращения к `os.environ` делают конфигурацию неотслеживаемой. Требование тикета — отсутствие обязательной переменной роняет приложение при старте громко, с указанием поля. Локально переменные живут в `.env`.

Альтернатива: самописный модуль на `os.environ` без новых зависимостей — плюс ~15 строк кода, ручная загрузка `.env`, ручная валидация и приведение типов.

## Решение

`pydantic-settings` (класс `Settings(BaseSettings)` в `app/config.py`):

- обязательные поля объявляются без дефолтов — валидация при создании, `ValidationError` с именем поля;
- `.env` читается библиотекой, без самописного кода;
- доступ через `get_settings()` с `lru_cache` — чтение один раз на процесс; функция, а не модульная константа, чтобы импорт модуля не требовал настроенного окружения (важно тестам).

Чтение `os.environ` где-либо кроме `app/config.py` запрещено.

## Последствия

- (+) Валидация и приведение типов бесплатно; библиотека из экосистемы FastAPI.
- (+) Тесты легко изолируются: `Settings(_env_file=None)` + monkeypatch окружения.
- (−) Две новые зависимости: `pydantic-settings`, `python-dotenv` (транзитивно).
- Новые настройки добавляются только полями `Settings` — иначе теряется «единственная точка».
