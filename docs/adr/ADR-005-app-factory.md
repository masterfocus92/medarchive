# ADR-005 — Фабрика приложения `create_app()` вместо модульного синглтона

## Статус

Принято владельцем 13.07.2026 (одобрение плана T1.2-BE).

## Контекст

FastAPI-приложение можно собирать при импорте модуля (модульный синглтон `app = FastAPI(); @app.get(...)`) или через фабрику. Впереди этап 2 (auth: приложению понадобятся session-механика и конфиг) и тесты, которым нужно собирать приложение с подменёнными настройками; синглтон намертво связывает сборку приложения с состоянием окружения на момент импорта.

## Решение

`app/main.py` экспортирует фабрику `create_app() -> FastAPI`; модульная переменная `app = create_app()` остаётся только как точка входа для uvicorn. Роуты живут в отдельных модулях `app/routes/*` и подключаются роутерами — роут не знает, как собирается приложение.

Фабрика не читает `get_settings()`: конфиг подключается там, где появляется первый реальный потребитель (сессия БД, T1.3+). Благодаря этому каркас стартует без docker и `.env` — а `/health` остаётся liveness-пробой без зависимости от БД.

## Последствия

- (+) Тесты собирают свежее приложение на каждый случай (`TestClient(create_app())`) — нет протекания состояния между тестами.
- (+) Когда конфиг станет нужен, он войдёт параметром фабрики (`create_app(settings)`) без переделки роутов.
- (−) На одну косвенность больше, чем в туториалах FastAPI; входная точка для uvicorn — `app.main:app`, а не «файл с приложением».
