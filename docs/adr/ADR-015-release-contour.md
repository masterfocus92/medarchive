# ADR-015 — Релизный контур: один VPS, два изолированных контура, ветки stg/main

## Статус

Принято владельцем 15.07.2026.

## Контекст

Проект дорос до первого релиза (этапы 1–6 закрыты). Владелец хочет: локальная разработка → релиз на тестовый стенд (ручное тестирование) → продакшн; стенд максимально приближен к проду; релизы не должны затрагивать данные прода; прод «не чувствует» жизни стенда.

Ресурсы: один VPS (Ubuntu 22.04, 1 core, 1 ГБ RAM, 30 ГБ диска), на нём уже живёт другая PWA владельца за **Caddy**. Домен продукта — **mdkarta.online** (новый, куплен под архив). Репозиторий — GitHub, CI пока нет.

## Решение

1. **Один VPS, два полностью изолированных контура** на нём:
   - `/opt/medarchive/prod` и `/opt/medarchive/stg` — независимые checkout'ы, свои `.env` (свои `SECRET_KEY`, свои ключи RouterAI, свои `FILES_DIR`), свои systemd-юниты (`medarchive-prod` :8000, `medarchive-stg` :8001).
   - Одна инсталляция PostgreSQL, **две БД**: `medcard_prod`, `medcard_stg`.
   - Отдельный VPS под прод отвергнут: для семейного POC изоляции сервисами/БД/каталогами достаточно, а близость стенда к проду (то же железо, ОС, версия PG, тот же Caddy) — максимальная.
2. **Домены**: `mdkarta.online` → prod, `stg.mdkarta.online` → stg. Оба — блоки в существующем Caddyfile (Caddy сам получает сертификаты); текущая PWA владельца не затрагивается. Несколько доменов на одном VPS — штатный режим реверс-прокси.
3. **Ветки**: `main` = прод, `stg` = стенд, разработка — в `feature/*`. На `stg` попадают только принятые владельцем и локально протестированные фичи; на `main` — **только merge из `stg`** (на прод едет ровно то, что тестировалось).
4. **Деплой** — один скрипт `deploy.sh <prod|stg>` на оба контура: checkout ветки контура → `uv sync --frozen` → `alembic upgrade head` (на БД контура) → рестарт юнита → smoke `/health`. Откат = деплой предыдущего коммита (+ `alembic downgrade`; обратимость миграций покрыта тестами).
5. **CI/CD — GitHub Actions**: на push/PR — ruff + полный pytest (Postgres+pgvector сервис-контейнером); push в `stg` → автодеплой стенда по SSH; push в `main` → деплой прода через GitHub Environment с **ручным подтверждением**.
6. **Гарантия «прод не чувствует релизов»**: стенд и его синк не касаются прод-БД, прод-файлов и прод-сервиса вообще (см. ADR-016 — стенд питается из бэкапа). Единственное, что меняет прод, — его собственный деплой: рестарт сервиса и alembic-миграции, к этому моменту отрепетированные на стенде на копии боевых данных. Данные записей деплой не трогает никогда.
7. **Стенд маркирован в UI**: `APP_ENV=stg` → плашка «Тестовый стенд» на всех экранах. Данные стенда неотличимы от боевых — без маркера реальная запись, внесённая на стенд по ошибке, погибнет при следующем синке.

## Последствия

- Рабочий процесс разработки меняется: этапы ведутся в `feature/stage-N`, после приёмки владельцем — merge в `stg`, после стендового тестирования — merge в `main`. История `main` становится историей продакшна.
- 1 ГБ RAM требует дисциплины: по одному uvicorn-воркеру на контур, swap, нативный PG (ADR-017); при нехватке памяти стг-сервис можно останавливать между релизами.
- Секреты живут в `.env` контуров на VPS (руками, вне git) и в GitHub Actions Secrets (SSH-ключ деплоя).
- Появляется `RELEASE.md`: процесс релиза и чек-лист ручного смоука стенда.
