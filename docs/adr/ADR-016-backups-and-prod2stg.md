# ADR-016 — Бэкапы в Yandex Object Storage; стенд восстанавливается из бэкапа

## Статус

Принято владельцем 15.07.2026.

## Контекст

Бэкапы — Must спеки (§8): лить реальные медицинские данные на прод без них нельзя. Владелец хочет ежедневный prod2stg-синк («стенд приближен к проду») плюс возможность запустить синк руками в любой момент; жёсткое требование — синк не должен касаться прода. Кандидаты хранилища: Яндекс или Рег.ру.

## Решение

1. **Хранилище — Yandex Object Storage** (S3-совместимое): зрелый S3-API под rclone/restic, холодный класс хранения, оплата по факту. Рег.ру отвергнут по принципиальной причине: если VPS у того же провайдера, бэкап разделяет судьбу сервера — а бэкап должен переживать в том числе проблемы провайдера.
2. **Ночной бэкап прода** (systemd-timer, ~03:00): `pg_dump medcard_prod` (custom-формат, сжатие) + архив `FILES_DIR` → загрузка в бакет через **rclone с crypt-remote** (медицинские данные в чужом облаке лежат только шифрованными; ключ шифрования — на VPS и в оффлайн-копии владельца, без него бэкап нечитаем). Ретенция: 14 ежедневных + 6 недельных; чистка — тем же скриптом.
3. **prod2stg — восстановление из последнего бэкапа**, не прямой дамп прода (systemd-timer ~04:00 + ручной запуск `prod2stg.sh` в любой момент):
   - скачать последний дамп и архив файлов из бакета → пересоздать `medcard_stg` → restore → `alembic upgrade head` кодом стенда → развернуть файлы в stg-`FILES_DIR` → рестарт стенда.
   - Прод в синке **не участвует вообще** — читается только бэкап. Направление строго prod→stg; механизма обратного переноса не существует по построению.
4. Побочные выгоды схемы, ради которых она и выбрана:
   - каждый синк — **ежедневная проверка восстановимости бэкапа** (мёртвый бэкап обнаружится на стенде, а не в день катастрофы);
   - каждый стендовый релиз — **репетиция прод-миграций на копии боевых данных**.

## Последствия

- Стенд отстаёт от прода максимум на сутки (или на минуты — после ручного `prod2stg.sh`).
- Данные, внесённые на стенде, живут до следующего синка — поэтому плашка «Тестовый стенд» обязательна (ADR-015).
- Ключ rclone-crypt становится критическим секретом: без него бэкапы нечитаемы; хранится на VPS и в оффлайн-копии у владельца.
- Восстановление прода при катастрофе — тем же механизмом, что prod2stg (отработан ежедневно), только в прод-БД.
