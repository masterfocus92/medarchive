# ADR-012 — Статусная модель записи: конвейер и подтверждение — две оси

## Статус

Принято владельцем 14.07.2026 (архитектурное обсуждение при приёмке этапа 3, T3.5-BE).

## Контекст

Первая версия (T3.1/T3.2) держала один статус `parse_status` со значением `confirmed` в конце конвейера. Владелец заметил: запись без файла со статусом `confirmed`, а с файлом — `uploaded`, — неочевидная логика, которую тяжело поддерживать. Разбор подтвердил два системных дефекта склейки:

1. Поле отвечало на два разных вопроса: техническое «что с разбором файлов» и продуктовое «проверил ли человек поля». Для записи без файла конвейера не существует — финальный статус несуществующего процесса читался как противоречие.
2. Подтверждение затирало `parsed`/`parse_failed` — терялись данные о качестве экстрактора (риск №2 OVERVIEW: качество AI на реальных документах надо мерить).

## Решение

Две независимые колонки в `health_records`:

- **`parse_status`** — только конвейер файлов: `none | uploaded | parsing | parsed | parse_failed`. `none` — файлов нет. Терминальные значения не изменяются никогда.
- **`confirmed_at`** (timestamptz, NULL) — момент подтверждения человеком; одностороннее. Запись без файлов подтверждается при создании (`= created_at`): автор и есть проверяющий. Запись с файлами — на экране проверки (Э4).

Предикаты потребителей: поиск (Э7) — `confirmed_at IS NOT NULL`; экран проверки — терминальный статус + не подтверждена; UI-лейбл — «подтверждено» при `confirmed_at`, иначе лейбл конвейера. Схема и таблица правил — `flows/record-creation.md` §6, `docs/code/README.md`.

## Последствия

- (+) Каждое поле отвечает на один вопрос; история конвейера сохраняется как датасет качества экстрактора.
- (+) Будущие сценарии (Э6 редактирование, добавление файла к заметке) не ломают семантику: оси движутся независимо.
- (−) Два поля вместо одного: потребители обязаны использовать предикаты из документации, а не изобретать свои комбинации.
- Миграция перенесла существующие `confirmed` без потерь. Урок побочный: миграции с данными тестируются на данных (тест `test_migration_c9d0_converts_confirmed_with_data`) — пустая БД не ловит ошибки порядка операций с constraint'ами.
