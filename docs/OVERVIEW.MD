# OVERVIEW — Семейная медкарта («внешняя память о здоровье»)

> Единый спек продукта (POC). Карта для самостоятельной разработки владельцем; AI выступает **ревьюером** (правила — `REVIEW.md`).
>
> **Статус:** Discovery закрыт полностью. Все продуктовые потоки спроектированы, стек зафиксирован. Открытых блокеров нет; один принятый дефолт помечен `[дефолт]` — можно оспорить до соответствующего этапа.
> **Контекст:** русскоязычные документы российских клиник. AI работает и отвечает на русском.
> **Актуализировано:** 13.07.2026.

---

## 1. Суть продукта

**Одной строкой:** надёжная внешняя память о здоровье семьи — чтобы нужный факт о здоровье можно было достать хоть через минуту, хоть через год.

**Фрейм:** личная (семейная) медкарта — единое место для всей информации о здоровье, чтобы уйти от бумаги и записей, разбросанных по разным клиникам.

**Стержневая проблема:** информация о здоровье не сохраняется в форме, пригодной для извлечения позже. Две грани боли:
- **Доступ в моменте** — нужный документ/факт недоступен, когда срочно нужен (перед приёмом, в острой ситуации).
- **Память во времени** — через год факт не вспомнить: что кололи, какая была реакция, какие были показатели.

**«Сквозность» в POC = ручная консолидация, не интеграция.** Каждая запись хранит клинику и врача; история собирается поверх всех клиник по конкретному человеку. Данные попадают в карту только руками (фото/скан + ручная фиксация). `[флаг MVP: интеграции ЕГИСЗ/ЕМИАС, МИС клиник, лаборатории — отдельный трек, в POC не входит, ядро на нём не строим]`

---

## 2. Пользователи

Одна семья: два **оператора** с учётными записями (Дмитрий, жена) и дочь — **пациент без учётки**. Все трое — пациенты. Ключевой пользователь для главной метрики — жена: продукт удался, если она вносит записи сама, без напоминаний.

---

## 3. Метрики успеха POC

Успех POC — поведенческий, снимается через 2–3 месяца обкатки:

1. **Накопление без принуждения** — новые записи появляются без самопринуждения и напоминаний.
2. **Реальное обращение в моменте** — продукт открыт перед приёмом / в нужной ситуации, и нужное найдено вместо старого способа.
3. **Вытеснение старого носителя** — прививки и развитие дочери ведутся здесь, а не в бумаге/заметках/голове.

**★★ Главный проектный принцип (следствие метрики, применяется везде):**
> **Минимизация трения при вводе — высший приоритет, выше полноты и структурированности данных.**
- Ввод максимально быстрый. Цель: «сфотографировал → запись существует».
- Ничего не обязательно, кроме абсолютного минимума. У всего есть дефолты.
- Структура наводится **лениво** (обогащение после ввода). Сырьё впускается любой ценой.

## 4. Границы (Scope)

### В POC
- Хранение документов (фото / PDF / файл).
- Профили семьи (трое) в одном пространстве.
- Запись как структурированный факт + опциональный комментарий.
- Развитие ребёнка (рост/вес/окружность головы) — через обычную запись без файла.
- Прививки — через обычную запись (сделанная = событие; будущая = текст в комментарии).
- Поиск/доступ по человеку, дате, типу.
- Сверхпростой ввод (фото → запись существует).
- **AI-экстрактор на вводе** — распознавание, авторазбор по полям, предзаполнение.
- **AI-поиск (строго retrieval)** — ответ на вопросы по карте, только извлечение существующих фактов, всегда со ссылкой на источник, без интерпретаций.

### Граница AI (зашита в требования)
> AI **извлекает и цитирует**, но **не интерпретирует и не советует**.
> Разделительная линия — между «найти твой факт» и «оценить твоё здоровье».
> Нет ответа в карте → честное «не нашёл», без выдумки.

### Вне POC (Won't) → бэклог (§13)
Интеграции с клиниками/лабораториями · Разграничение доступа между операторами · AI-интерпретация анализов и медсоветы · Push-напоминания · Расшаривание врачу · Экспорт данных · Граф динамики роста/веса · Сменный AI-провайдер / MCP · Роль «няня/родственник» · Ролевая модель и права.

## 5. Доменная модель

**Агрегат:** `Семья → Член семьи → Запись о здоровье`

### Семья
Корень агрегата (семейное пространство). Содержит членов семьи.

### Член семьи
| Атрибут | Примечание |
|---------|-----------|
| `ФИО` | |
| `дата_рождения` | Возраст на момент записи — важно для младенца и будущего reasoning |
| `пол` | Для будущего reasoning (нормы зависят от пола/возраста) |
| `имеет_учётную_запись` | да (логинится, может быть автором и пациентом) / нет (только пациент) |
| `админ` | Управляет составом семьи. В POC оба взрослых = админы. Без учётки → не админ по определению |

### Запись о здоровье

**Системные (проставляются сами):**
- `дата_создания` — момент внесения, всегда «сейчас».
- `автор` → **только** член семьи с учётной записью. **[инвариант: ребёнок без учётки не может быть автором]**

**Обязательное:**
- `пациент` → любой член семьи. AI определяет из документа; выбор из членов семьи (один тап) как fallback.
- **[инвариант: запись содержит хотя бы один файл ИЛИ непустой комментарий. Пустых записей не существует]**

**Опциональные** (AI предлагает, человек правит, ничто не блокирует сохранение):
- `файлы` — **упорядоченная коллекция** (0..N), фото/PDF. Порядок значим (страницы). AI трактует коллекцию как единый документ → один набор полей.
- `дата_события` — реальная дата документа/события (≠ дата создания). Будущие события — только текстом в комментарии.
- `название`.
- `клиника` — структурное, фильтруемое.
- `врач` — структурное, фильтруемое.
- `тип` — **открытое поле**: AI предлагает, частые варианты для быстрого выбора, можно вписать своё или оставить пустым.
- `содержание` — свободный текст из документа; **индексируется для поиска**.
- `комментарий` — заметка человека («карандаш»); **индексируется для поиска**.

**Мягкое удаление:** `удалена` (+ когда, кем). Физически не стирается.

## 6. Спроектированные потоки (ранее открытые — все закрыты)

**Вход.** Email + пароль, **две предзаведённые учётки** (создаются seed-скриптом, без регистрации и восстановления пароля — «спроси Дмитрия»). Долгоживущая сессия: вход — одноразовый акт на устройство. Две учётки обязательны — иначе `автор` бессмыслен.

**Онбординг семьи.** Seed-скрипт: учётки, семья, три члена семьи (ФИО, дата рождения, пол). Без UI.

**Ввод: файл первичен.** Загрузка → файл сохранён и запись существует **до** всякого разбора. Провал AI-разбора не отменяет сохранение. Статусы записи: `загружено` → `разбирается` → `разобрано` / `разбор не удался` — все состояния видны в UI. «Сохранено» (мгновенно) и «разобрано» (асинхронно) — разделены. Детальный путь со всеми ветвлениями (камера/файлы, экраны, события) — `flows/record-creation.md` (утверждён 14.07.2026).

**Просмотр.** Лента записей по профилю (удалённые отфильтрованы по умолчанию, на уровне репозитория) → карточка записи со всеми полями, пролистывание многостраничного файла, открытие/скачивание оригинала. Переключатель профилей — монограммы (дизайн-кит).

**Редактирование.** `[дефолт, можно оспорить до этапа 6]` Тот же экран проверки, что после AI-разбора, открытый в режиме правки. Один экран на два сценария — дёшево и консистентно.

**Удаление.** Только после подтверждения. Soft delete с каскадной дисциплиной: все выборки по умолчанию исключают удалённые (лента, профиль, любой список); удалённая запись выпадает из поискового индекса; файлы остаются на диске, но недоступны из UI.

**Поиск.** Векторный (семантика, «агент понимает, что ищет»). Эмбеддинги генерируются при подтверждении записи из `содержание` + `комментарий` + структурных полей. Выдача — всегда со ссылкой на запись-источник. Экстрактор и поиск — одна цепочка: качество разбора на входе определяет точность поиска.

## 7. User Story Map

Роль по умолчанию — **Оператор**.

**Активность 1 — Начало работы (seed, без UI):** учётки · вход · семья · члены семьи.

**Активность 2 — Внести запись:** сфотографировать · загрузить PDF · несколько файлов в одну запись · AI-разбор по полям · AI-определение пациента · подтверждение одним действием · правка поля · выбор пациента вручную (fallback) · сохранение при провале AI · запись без файла (комментарий) · рост/вес как запись.

**Активность 3 — Найти и вспомнить:** лента по человеку · переключение профилей · вопрос AI словами · ответ только по карте + ссылка · честное «не нашёл» · фильтры ленты `[Could]`.

**Активность 4 — Просмотр и работа с записью:** открыть запись · пролистать многостраничный файл · открыть/скачать оригинал · редактировать · удалить (soft, с подтверждением).

**Активность 5 — Надёжность:** автоматическое резервирование `[Must]`.

**Backbone:** `А1` → `А2` → `А3` → `А5`. Без любой из четырёх POC не живёт.

**Кандидаты на срез из первого релиза:** фильтры ленты · AI-поиск (лента сама даёт ценность — поиск добить вторым).

## 8. Приоритизация (MoSCoW)

**Must:** загрузка записи (фото/PDF/файл) · AI-экстрактор с предзаполнением · сохранение в один тап, правка любого поля в один тап · отказоустойчивость экстрактора (провал не блокирует) · разделение «сохранено»/«разобрано» · профили семьи · лента + открытие файла · комментарий · **резервное копирование**.

**Should:** AI-поиск (retrieval, ответ + ссылка) · прививки и рост/вес через обычную запись · редактирование записи.

**Could:** фильтры ленты (тип/дата/клиника) · визит-без-файла как явный поток.

**Won't (POC):** см. §4.

## 9. Нефункциональные требования

**Платформа:** PWA — server-rendered web-приложение с PWA-манифестом (иконка на телефоне, камера и загрузка файла из коробки, без апсторов). `[флаг MVP: нативное приложение при упоре в ограничения PWA]`

**Хранение (POC):** всё на собственном VPS — Postgres (структура + векторный индекс) + файлы на локальном диске. Синхронизация — сервер общий, жене просто ссылка. `[флаг MVP: managed-провайдеры]`

**Приватность (осознанный размен POC):** содержимое меддокументов уходит внешним AI-провайдерам (экстрактор, эмбеддинги, поиск). Принято владельцем как цена за магию. `[флаг MVP: сменный провайдер / локальная модель; 152-ФЗ и медтайна — при выходе на чужие семьи]`

**Безопасность (минимум POC):** доступ только аутентифицированный; файлы и записи недоступны по прямой ссылке без авторизации.

**Резервное копирование — Must, условие запуска обкатки:** автоматический регулярный бэкап **обоих** видов данных (БД и файлы) + **проверяемое восстановление** — непроверенный бэкап = отсутствие бэкапа.

## 11. Процесс разработки

**Роли:** владелец пишет код; AI — ревьюер (правила и чек-лист — `REVIEW.md`).

**Цикл на этап:** формулировка задачи → AI проверяет декомпозицию до кода → разработка мелкими коммитами → ревью diff по чек-листу → фикс блокеров → merge.

**Этапы (вертикальные срезы):** 1. Каркас и БД → 2. Вход и профили → 3. Ввод (файл первичен) → 4. AI-разбор → 5. Лента и карточка → 6. Удаление и редактирование → 7. Поиск → 8. Обкатка (бэкапы, деплой, перенос реальных документов).

## 12. Риски

| Риск | Суть | Митигация |
|------|------|-----------|
| **Нет второй пары глаз** | Код пишется в одиночку: слепые зоны, эрозия инвариантов, «работает — и ладно» | Обязательное AI-ревью каждого этапа по `REVIEW.md`; мелкие коммиты; ревью декомпозиции до кода |
| **AI-экстрактор на русских медданных** | Рукописи, печати, кривые фото → низкое качество разбора → опыт хуже ручного ввода → удар по главной метрике | Проверять рано (этап 4, до ленты), на **реальных** документах семьи, не на синтетике |
| **Трение прокрадётся** | Лишние поля и подтверждения «для полноты» | Каждая фича проверяется вопросом «приближает к метрике или нет»; безжалостно резать |
| **Приватность через LLM** | Для POC принято; на MVP — юридическая стена (152-ФЗ) | Флаг MVP; сменный провайдер |
| **Ресурс владельца** | Работа + семья; самостоятельная разработка дольше, чем ревью агента — риск заглохнуть вырос | Безжалостно маленький POC; вертикальные срезы — живой результат на каждом этапе |

## 13. Бэклог (вне POC)

- Разграничение доступа между операторами; приватные зоны.
- Роль «няня/родственник» — внешний член с ограниченными правами.
- Интеграции: ЕГИСЗ/ЕМИАС, МИС клиник, лаборатории (Инвитро, Хеликс).
- AI-интерпретация анализов и медсоветы (отдельная боль: понимание).
- Push-напоминания о визитах и прививках.
- Граф динамики роста/веса поверх записей-событий.
- Сменные модели и плоскости развёртывания (обсуждено 14.07.2026): self-hosted дистрибутив (весь data plane — файлы, БД, векторы, модели — у пользователя; ближайший к текущей архитектуре путь) · SaaS с BYO-ключами LLM/эмбеддера · гибрид (data plane у клиента, control plane как сервис) — только под платящего B2B-клиента · MCP. Уже действующие обязательства под это: AI за доменным интерфейсом (ADR-013), эмбеддинги хранятся с именем и версией модели (этап 7).
- Расшаривание данных врачу. Экспорт данных.
- Фильтры ленты — первый кандидат на добавление после обкатки.
