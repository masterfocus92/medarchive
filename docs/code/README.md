## 2. Структура проекта

```
app/                    веб-приложение
  main.py               фабрика create_app(settings) — запуск только factory-режимом (ADR-005)
  config.py             конфигурация — единственная точка чтения окружения (ADR-001)
  middleware.py          default-deny: всё непубличное требует сессию (ADR-010)
  routes/               HTTP-роуты, подключаются в фабрике роутерами
    system.py           GET /health — liveness-проба (без обращения к БД)
    pages.py            GET / — главная; здесь же настроен Jinja2
    auth.py             GET/POST /login, POST /logout
    profiles.py         POST /profile/{id} — переключение активного профиля
    deps.py             get_current_account — учётка текущего оператора
  seed.py               наполнение БД стартовыми данными (данные — из .env.seed, ADR-009)
  services/

    index.html          главная: персонализированное пустое состояние
    login.html          вход на примитивах кита (field + btn-primary)
    macros/ui.html      примитивы кита как Jinja-макросы (один примитив — одно место)
  static/               статика, отдаётся на /static
    css/tokens.css      токены — ДОСЛОВНАЯ копия :root кита; менять только через кит
    css/app.css         базовый слой + @font-face + CSS использованных примитивов
    fonts/              woff2 self-hosted: Inter, Source Serif 4, JetBrains Mono (ADR-008)

  test_config.py        тесты конфигурации
  test_app.py           тесты каркаса: /health, рендер /, маунт статики

docs/
```

Устройство приложения: `create_app(settings)` собирает FastAPI-приложение (маунт `/static`, роутеры, middleware сессии и default-deny); роуты не знают, как собирается приложение, и живут в `app/routes/*`. Настройки передаются фабрике параметром (тесты собирают приложение без `.env`); импорт модуля окружения не требует — поэтому запуск только factory-режимом. Пути к шаблонам и статике привязаны к положению пакета, а не к cwd.

## 3. Конфигурация

Все настройки — через `app/config.py` (`get_settings()`); чтение `os.environ` в остальном коде запрещено (ADR-001). Отсутствие обязательной переменной роняет приложение при старте с указанием поля.

| Переменная | Обязательна | Что это |
|---|---|---|
| `DATABASE_URL` | да | Подключение к Postgres, формат `postgresql+psycopg://user:pass@host:5432/db` |
| `FILES_DIR` | да | Каталог хранения файлов записей (в git не попадает) |
| `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` | нет (дефолт `medcard`) | Креды контейнера БД, читает docker compose; должны совпадать с `DATABASE_URL` |
| `SECRET_KEY` | да (≥ 32 символов) | Подпись session-cookie. У каждой машины свой; генерация — команда в `.env.example` |
| `SEED_*` (19 полей, файл `.env.seed`) | для seed | Данные семьи: фамилия/имя/отчество (опц.)/дата/пол троих, email и пароли взрослых. Класс `SeedSettings`. **Вне git** (ADR-009), шаблон — `.env.seed.example` |

Локально переменные живут в `.env` (создаётся как `cp .env.example .env`, в git не попадает).

## 4. Окружение и команды

Подъём с нуля — `README.md` в корне (5 шагов). Ежедневные команды:

| Команда | Что делает |
|---|---|
| `uv run uvicorn app.main:create_app --factory --reload` | Запустить приложение — http://localhost:8000 |
| `docker compose up -d` | Поднять БД (данные в named volume `pgdata`, переживают пересоздание) |
| `uv sync` | Установить/обновить зависимости по `uv.lock` |
| `uv run pytest` | Прогнать тесты |
| `uv run ruff check .` / `uv run ruff format .` | Линт / формат вручную |
| `uv run pre-commit install` | Включить хуки на коммит (после — линт запускается сам) |
| `uv run alembic upgrade head` | Накатить миграции (нужен `.env` и поднятая БД) |
| `uv run python -m app.seed` | Создать семью и учётки по `.env.seed` (повторный запуск безвреден) |
| `uv run alembic revision --autogenerate -m "..."` | Сгенерировать миграцию по изменениям моделей — проверять глазами |

Все команды запускаются из корня проекта — пакет `app` не устанавливается в venv (ADR-004).

## 5. Сущности

Агрегат: семья → член семьи → запись о здоровье (`OVERVIEW.MD` §5). Схема создаётся только миграциями Alembic.

| Таблица | Модель | Суть |
|---|---|---|
| `families` | `Family` | Семейное пространство. **Название обязательное; продуктовое решение владельца (13.07.2026): дефолт — «Семья»** — семью можно создать, не придумывая имя, но безымянной она не бывает |
| `family_members` | `FamilyMember` | Член семьи — всегда пациент. Фамилия и имя обязательны, **отчество опционально** (T2.7; `full_name` — свойство для отображения), дата рождения, пол (CHECK `male/female`). **Учётка строго опциональна (0..1): у ребёнка её может НЕ быть** — это норма, ничто в схеме учётку не требует |
| `accounts` | `Account` | Учётная запись оператора: email (unique), хэш пароля, `is_admin`. Одна учётка на человека (unique FK). Авторство и админство — свойства учётки (ADR-006): «автор без учётки» и «админ без учётки» невыразимы в схеме. Состав семьи меняют админы |
| `health_records` | `HealthRecord` | Запись: автор (FK → `accounts`), пациент (FK → `family_members`), `created_at` проставляет БД (записи «задним числом» невозможны). Опциональные поля ничего не блокируют: дата события, название, клиника, врач, тип (открытая строка, не enum), содержание, комментарий. Soft delete: `deleted_at` + `deleted_by_account_id`, физически не стирается |
| `record_files` | `RecordFile` | Файлы записи, порядок значим (страницы): unique `(record_id, position)`. Путь — относительно `FILES_DIR` |

Инвариант «запись содержит файл ИЛИ комментарий» на уровне СУБД не выразим (две таблицы) — его обеспечит сервисный слой создания записи (этап 3).

## 6. Сценарии

| Маршрут | Доступ | Что делает |
|---|---|---|
| `GET /` | сессия | Главная: шапка с переключателем профилей и выходом; пустое состояние «{Имя} — записей пока нет» (первичное действие добавит этап 3) |
| `GET /login` | публичный | Форма входа |
| `POST /login` | публичный | Проверка пары email+пароль → сессия на 180 дней → редирект на `/`; отказ — та же форма с ошибкой, email сохранён |
| `POST /logout` | любой | Сброс сессии (включая выбор профиля) → редирект на `/login`; без сессии просто редиректит |
| `POST /profile/{member_id}` | сессия | Переключить активный профиль (член своей семьи) → 303 на `/`; чужой и несуществующий id неразличимы (404) |
| `GET /health` | публичный | Liveness-проба процесса: `{"status": "ok"}`, БД сознательно не проверяет |
| `GET /static/*` | публичный | Статика приложения (стили, шрифты). Файлы записей будут раздаваться НЕ отсюда |

Всё, чего нет в списке публичного (`app/middleware.py`), закрыто по умолчанию. Продуктовые сценарии (загрузка, лента) начнутся с этапа 3.

## 7. Тесты

- `tests/test_config.py` — конфигурация: отсутствие `DATABASE_URL` или `FILES_DIR` даёт `ValidationError` с именем поля; валидное окружение читается. Тесты изолированы от реального `.env` разработчика (фикстура чистит окружение, `_env_file=None`).
- `tests/test_app.py` — каркас и дизайн-слой: `/health` отвечает ровно `{"status": "ok"}`; `/` отдаёт пустое состояние (`empty`, «Записей пока нет»); CSS отдаётся с 200; `/static` обслуживается приложением. **Тесты-стражи дизайн-системы:** во всех шаблонах нет сырых hex-цветов и inline-стилей, в CSS экранов (кроме `tokens.css`) нет hex — экран с сырым значением уронит тесты. БД и `.env` не требуют.
- `tests/test_models.py` — интеграционные, на реальном Postgres из docker (отдельные БД `medcard_test*`, основная не трогается; без docker — skip с подсказкой; общая инфраструктура — `conftest.py`): цикл миграции up→down→up без остатка; дефолт названия семьи; член семьи без учётки валиден; уникальность email и «одна учётка на человека»; `created_at` ставит БД; дубль позиции файла режется constraint'ом.
- `tests/test_seed.py` — seed на тестовых данных: структура (1 семья, 3 члена, 2 админ-учётки, дочь без учётки), пароль хранится bcrypt-хэшем, идемпотентность; негативные: без данных и с пустым паролем — `ValidationError` до обращения к БД.
- `tests/test_auth.py` — вход/сессия/защита: успешный вход открывает главную (сюда переехали тесты её рендера), cookie с Max-Age; неверный пароль и несуществующий email неразличимы байт-в-байт; `/` без сессии редиректит, публичные пути доступны; logout очищает сессию и не падает без неё.
- `tests/test_login_page.py` — экран входа (T2.3-FE): примитивы кита на месте (field, одна btn-primary), автофокус/autocomplete, ошибка с `role="alert"` и сохранённым email; страж не находит сырых стилей.
- `tests/test_profiles.py` — профили (T2.4 + T2.5): юниты выбора активного (дефолт — оператор, выбор из сессии, мусорный id → дефолт без ошибки) и инициалов; интеграционно (две семьи в тестовой БД): свой член → 303, чужой/несуществующий → 404, без сессии → на `/login`; переключатель — три монограммы и ровно один `aria-selected`, пометка переезжает после переключения, пустое состояние меняет имя, на `/login` шапки нет.

Тестовый клиент — `fastapi.testclient.TestClient` (dev-зависимость `httpx2` — по требованию starlette, связка со старым `httpx` объявлена устаревшей).
