## 1. Что уже работает

- **Релизный контур — работает** (Э6.5 принят целиком 19.07.2026, ADR-015/016/017): прод `mdkarta.online` и стенд `stg.mdkarta.online` на одном VPS (нативный PG16, systemd, Caddy); плашка `env-banner` при `APP_ENV=stg`; ночной шифрованный бэкап в Yandex Object Storage (03:00) и пересборка стенда из бэкапа (04:00, ручной запуск `medarchive-prod2stg`); CI: пуш в `stg` → тесты → автодеплой стенда, PR `stg→main` + Approve → прод. Рабочий процесс: этапы в `feature/stage-N`, после приёмки — merge в `stg`; в прод владелец заливает сам (`RELEASE.md`). Runbook установки — `infra/README.md`.
- **Правка и удаление** (T6.1–T6.3, по `../flows/record-editing-deletion.md`): с карточки — `rec-actions`: «Редактировать» (`GET /records/{id}/edit` — тот же экран проверки в режиме правки: без AI-кодов и ретрая, «Сохранить» возвращает на карточку через `return_to=card` по белому списку, `confirmed_at` не переписывается — ADR-012) и «Удалить» (красным, обособленно). Удаление — только через страницу подтверждения («лист» кита, работает без JS): `soft_delete` — единственная точка (`deleted_at` + `deleted_by_account_id`), физически ничего не стирается, файлы и `extraction_runs` остаются. Каскадная дисциплина — фильтрами по умолчанию: удалённая исчезает из ленты, все её URL (карточка/edit/delete/файлы) — 404, неотличимо от несуществующей; повторный POST безопасен. «Удалить» доступно и с экрана проверки (мусорный скан). Восстановление — вручную в БД (Won't).
- **Дизайн-кит v2 — основной** (T5.5.1–T5.5.7, Э5.5): материал (тени `--lift-sm/md`, фактура `paper-tex` из токенов `--tex-*`), цветные статус-бейджи `badge` — pending с пульсом («загружено»/«разбирается») / review («разобрано — проверьте») / failed / done («подтверждено» — маркер карточки); код «AI подставил» — `field.ai` + `ai-hint` (список полей выводится из имеющихся данных без миграций: до подтверждения draft-поля наполняет только конвейер); предложение AI-пациента — баннер `ai-suggest` без кнопки, принятие — тап по монограмме; первичное действие каждого экрана — в нижнем `action-bar` (disabled «Сохранить» — только JS-зеркало B4: без JS кнопка активна, инвариант держит сервер); превью страниц формы — `pages`/`page-thumb`/`page-add` (клиентский накопитель, серверного staging нет, htmx не вводился — ADR-011 в силе); персональные акценты `--accent-1/2/3` по порядку членов семьи (активный профиль, заголовок ленты, выбранный пациент на формах). Вся логика отображения — `services/ui.py`; кит self-hosted, v1 — в истории git.
- **Лента и карточка записи** (T5.1–T5.4, по `../flows/record-viewing.md`): главная — лента активного профиля с двумя сортировками (`?sort=created` — хроника внесения, дефолт; `?sort=event` — медицинская хронология, записи без даты события встают по дате внесения; невалидное значение молча падает в дефолт) и заголовком-именем активного профиля. Элемент ленты — `rec-lite` (название serif, дата mono, чип типа, клиника), весь элемент — ссылка на `GET /records/{id}`; неподтверждённые несут статус-бейдж. Роут записи ветвится: подтверждённая → карточка просмотра `records/view.html` (read-only, полная композиция `record`: факты чернилами, заметка карандашом, маркер «подтверждено» в шапке), неподтверждённая → экран проверки. Страницы — вертикальным потоком `page-flow` внутри `rec-body` (`loading="lazy"`, подпись «стр. N из M», тап — оригинал в новой вкладке, «Скачать оригинал» — `<a download>`); PDF — плашка «открыть» без серверного превью; пропавший с диска файл — плашка «файл недоступен» + warning-лог, не 500. Временные счётчик и блок «Ждут проверки» упразднены.
- **AI-разбор** (T4.1–T4.5, ADR-013/014): после сохранения записи с файлами фоновый конвейер (`uploaded → parsing → parsed | parse_failed`) прогоняет страницы через RouterAI (`anthropic/claude-sonnet-5`, OpenAI-совместимый API) и кладёт черновик **только в пустые поля**; предложение пациента — отдельно от выбора человека. Экран проверки (неподтверждённые записи): статус текстом, meta-refresh при активном конвейере, правка полей, принятие AI-пациента одним тапом (пунктирная монограмма), «Сохранить» → `confirmed_at` (ставится один раз), «Разобрать ещё раз» при провале/зависании. Вход на проверку — статус-бейдж в ленте (с Э5). Каждый прогон — строка в `extraction_runs` (датасет качества). Ручная проверка качества: `uv run python -m app.tools.try_extractor <файлы>`.
- **Создание записи — «сфотографировал → запись существует»** (T3.1–T3.5, по `../flows/record-creation.md`): экран добавления с двумя потоками контента (камера постранично / галерея-файлы мультивыбором, смешивание в один список страниц — первый JS проекта, ADR-011, с полной деградацией без JS); сервис создания — единственная точка рождения записей: инвариант «файл ИЛИ комментарий», атомарность БД+диск с компенсацией, валидация типов по сигнатурам (jpeg/png/webp/heic/pdf, ≤20 МБ); статусная модель из двух осей (ADR-012, §5 ниже); тост на главной (счётчик записей Э3 заменён лентой в Э5); файлы раздаются `GET /records/{id}/files/{pos}` только своей семье.
- **Переключатель профилей и выход** (T2.5-FE): шапка `app-head` на каждом экране с данными — три монограммы (инициалы «имя+фамилия»), активный помечен `aria-selected`, переключение обычной формой без JS; «Выйти» обособлен справа. Пустое состояние персонализировано именем активного профиля (именительный падеж через тире — русская морфология без словаря не склоняется). На `/login` шапки нет.
- **Активный профиль** (T2.4-BE): «на кого я смотрю» живёт в сессии; дефолт — сам вошедший; `POST /profile/{id}` переключает (чужой/несуществующий id → 404, неразличимы). Контекст `/` отдаёт членов семьи с инициалами для переключателя (рендер — T2.5-FE). После logout выбор сбрасывается на дефолт — осознанно.
- **ФИО нормализовано** (T2.7-BE): `last_name`/`first_name` обязательные, `middle_name` опционально; миграция разделила существующие данные без потерь; `full_name` — свойство для отображения; инициалы без парсинга строк.
- **Экран входа на дизайн-системе** (T2.3-FE, merge PR #1): примитивы кита `field` и `btn-primary btn-lg` (тач-цель ≥ 44px), автофокус и autocomplete для менеджеров паролей; ошибка формы не привязана к полю (не подсказывает, что именно неверно). Hover primary-кнопки сознательно опущен: в ките он сырым hex, токена нет, продукт touch-first.
- **Вход, сессия, выход, закрытость по умолчанию** (T2.2-BE): email+пароль → подписанная cookie на 180 дней (ADR-010); `POST /logout`; всё, кроме `/login`, `/health`, `/static/*`, без сессии редиректит на вход (default-deny middleware). Отказы «нет email» и «неверный пароль» неразличимы снаружи, включая время ответа.
- **Seed семейного пространства** (T2.1-BE + T2.6-BE): `uv run python -m app.seed` создаёт семью, трёх членов и две админ-учётки. Идемпотентен («любая семья уже есть» = уже засеяно). Сообщения честные: ошибка валидации называет настоящую причину по каждому полю; плейсхолдеры шаблона в данных — отказ до записи; при «данные уже есть» БД сверяется с файлом и расхождения перечисляются по полям **без значений**. Все реальные данные — в `.env.seed` вне git (ADR-009), в репозитории только шаблон.
- **Layout на дизайн-системе** (T1.4-FE): главная рендерится на токенах кита (`tokens.css` — копия `:root` из `docs/design-kit.html`), self-hosted шрифты (ADR-008), пустое состояние на примитиве `empty` через Jinja-макрос. Тест-страж не пускает сырые цвета и inline-стили в шаблоны.
- **Доменная схема БД** (T1.3-BE): пять таблиц (семья, члены, учётки, записи, файлы записей) с инвариантами на уровне схемы; миграции Alembic воспроизводимы в обе стороны.
- **Веб-приложение запускается** (T1.2-BE): `uv run uvicorn app.main:create_app --factory --reload` → главная и liveness-проба `/health`.
- Инфраструктура разработки (T1.1-INFRA): локальная БД в docker (PostgreSQL 16 + pgvector), модуль конфигурации с валидацией при старте, тулинг качества (Ruff, pytest, pre-commit).

Следующий этап — 7 «Поиск» (эмбеддинги + pgvector; не нарезан; по правилу `docs/flows/` — сначала поток на утверждение). Обязательства Э7, зафиксированные потоками: эмбеддинги при подтверждении и пересчёт при правке; удалённые не в индексе, при удалении — чистка вектора.

Пути к шаблонам и статике привязаны к положению пакета, а не к cwd.

## 3. Конфигурация

Все настройки — через `app/config.py` (`get_settings()`); чтение `os.environ` в остальном коде запрещено (ADR-001). Отсутствие обязательной переменной роняет приложение при старте с указанием поля.

| Переменная | Обязательна | Что это |
|---|---|---|
| `DATABASE_URL` | да | Подключение к Postgres, формат `postgresql+psycopg://user:pass@host:5432/db` |
| `FILES_DIR` | да | Каталог хранения файлов записей (в git не попадает) |
| `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` | нет (дефолт `medcard`) | Креды контейнера БД, читает docker compose; должны совпадать с `DATABASE_URL` |
| `SECRET_KEY` | да (≥ 32 символов) | Подпись session-cookie. У каждой машины свой; генерация — команда в `.env.example` |
| `APP_ENV` | нет (дефолт `dev`) | `dev` \| `stg` \| `prod` (ADR-015). При `stg` — плашка «Тестовый стенд» на каждом экране: данные стенда перезаписываются ночным prod2stg |
| `EXTRACTOR_PROVIDER` | нет (дефолт `disabled`) | `openai_compatible` \| `claude` (бэклог) \| `disabled`. При включённом — три поля ниже обязательны (иначе не стартует) |
| `EXTRACTOR_BASE_URL` / `EXTRACTOR_MODEL` / `EXTRACTOR_API_KEY` | при включённом провайдере | RouterAI: `https://routerai.ru/api/v1` + `anthropic/claude-sonnet-5` (ADR-014). Ключ в git не попадает |
| `SEED_*` (19 полей, файл `.env.seed`) | для seed | Данные семьи: фамилия/имя/отчество (опц.)/дата/пол троих, email и пароли взрослых. Класс `SeedSettings`. **Вне git** (ADR-009), шаблон — `.env.seed.example` |

Локально переменные живут в `.env` (создаётся как `cp .env.example .env`, в git не попадает).

## 4. Окружение и команды

Подъём с нуля — `README.md` в корне (5 шагов). Ежедневные команды:

| Команда | Что делает |
|---|---|
| `uv run uvicorn app.main:create_app --factory --reload` | Запустить приложение — http://localhost:8000 |
| `docker compose up -d` | Поднять БД (данные в named volume `pgdata`, переживают пересоздание) |
| `uv sync` | Установить/обновить зависимости по `uv.lock` |
| `uv run pytest` | Прогнать тесты |
| `uv run ruff check .` / `uv run ruff format .` | Линт / формат вручную |
| `uv run pre-commit install` | Включить хуки на коммит (после — линт запускается сам) |
| `uv run alembic upgrade head` | Накатить миграции (нужен `.env` и поднятая БД) |
| `uv run python -m app.seed` | Создать семью и учётки по `.env.seed`; повторный запуск безвреден и сверяет БД с файлом; плейсхолдеры шаблона — отказ |
| `uv run python -m app.tools.try_extractor <файлы>` | Прогнать реальные документы через экстрактор, результат в терминал (проверка качества) |
| `uv run alembic revision --autogenerate -m "..."` | Сгенерировать миграцию по изменениям моделей — проверять глазами |

Все команды запускаются из корня проекта — пакет `app` не устанавливается в venv (ADR-004).

## 5. Сущности

Агрегат: семья → член семьи → запись о здоровье (`OVERVIEW.MD` §5). Схема создаётся только миграциями Alembic.

| Таблица | Модель | Суть |
|---|---|---|
| `families` | `Family` | Семейное пространство. **Название обязательное; продуктовое решение владельца (13.07.2026): дефолт — «Семья»** — семью можно создать, не придумывая имя, но безымянной она не бывает |
| `family_members` | `FamilyMember` | Член семьи — всегда пациент. Фамилия и имя обязательны, **отчество опционально** (T2.7; `full_name` — свойство для отображения), дата рождения, пол (CHECK `male/female`). **Учётка строго опциональна (0..1): у ребёнка её может НЕ быть** — это норма, ничто в схеме учётку не требует |
| `accounts` | `Account` | Учётная запись оператора: email (unique), хэш пароля, `is_admin`. Одна учётка на человека (unique FK). Авторство и админство — свойства учётки (ADR-006): «автор без учётки» и «админ без учётки» невыразимы в схеме. Состав семьи меняют админы |
| `health_records` | `HealthRecord` | Запись: автор (FK → `accounts`), пациент (FK → `family_members`), `created_at` проставляет БД (записи «задним числом» невозможны). Опциональные поля ничего не блокируют: дата события, название, клиника, врач, тип (открытая строка, не enum), содержание, комментарий. Soft delete: `deleted_at` + `deleted_by_account_id`, физически не стирается |
| `record_files` | `RecordFile` | Файлы записи, порядок значим (страницы): unique `(record_id, position)`. Путь — относительно `FILES_DIR` |
| `extraction_runs` | `ExtractionRun` | Журнал прогонов разбора (включая ретраи) — навсегда: отладка + датасет качества экстрактора. `raw_response` — артефакт провайдера, непереносим (ADR-013) |

Инвариант «запись содержит файл ИЛИ комментарий» на уровне СУБД не выразим (две таблицы) — его обеспечивает сервис `services/records.create_record` (единственная точка создания записей).

### Статусная модель записи (ADR-012)

Две независимые оси; смешивать их или изобретать свои комбинации предикатов нельзя — только правила из таблицы ниже. Схема переходов — `../flows/record-creation.md` §6.

**Ось 1 — `parse_status` (конвейер разбора файлов, и только он):**

| Значение | Смысл | Кто ставит |
|---|---|---|
| `none` | Файлов нет — конвейера не существует. Терминальный | сервис при создании записи без файлов |
| `uploaded` | Файлы сохранены, разбор не начат | сервис при создании записи с файлами |
| `parsing` | Фон взял в работу | конвейер (Э4) |
| `parsed` | Черновик полей готов. **Терминальный, не затирается** — датасет качества экстрактора | конвейер (Э4) |
| `parse_failed` | AI упал; сохранение не отменялось (инвариант). **Терминальный** | конвейер (Э4) |

**Ось 2 — `confirmed_at` (подтверждение человеком):** NULL → timestamp, одностороннее. Запись без файлов подтверждена при создании (= `created_at`, автор и есть проверяющий); с файлами — после «Сохранить» на экране проверки (Э4), из любого терминального статуса (`parsed` — принял черновик, `parse_failed` — заполнил руками).

**Правила потребителей:**

| Вопрос | Предикат |
|---|---|
| Что попадает в поиск (Э7) | `confirmed_at IS NOT NULL` |
| Что ждёт проверки человеком | `parse_status IN ('parsed','parse_failed') AND confirmed_at IS NULL` |
| Что показать в UI | `confirmed_at` есть → «подтверждено» (`CONFIRMED_LABEL`); иначе `PARSE_STATUS_LABELS[parse_status]`; для `none` — ничего. Вариант бейджа кита v2 — `PARSE_STATUS_KINDS` (pending/review/failed) + `CONFIRMED_KIND` (done); правило — `services/ui.badge_for`. В ленте бейдж — только у неподтверждённых (`feed_badge`): норма не маркируется |
| Что видно в ленте | всё, кроме soft-deleted, в любом статусе; только активный профиль |

Русские подписи статусов и варианты бейджей — только из `app/models/record.py` (`PARSE_STATUS_LABELS`, `PARSE_STATUS_KINDS`, `CONFIRMED_LABEL`, `CONFIRMED_KIND`), в шаблонах не дублируются. Подпись parsed — «разобрано — проверьте» (решение 15.07.2026).

## 6. Сценарии

| Маршрут | Доступ | Что делает |
|---|---|---|
| `GET /` | сессия | Лента активного профиля: `?sort=created` (дефолт, хроника внесения) \| `?sort=event` (по дате события, fallback — дата внесения); бейджи у неподтверждённых; пустое состояние «{Имя} — записей пока нет» |
| `GET /login` | публичный | Форма входа |
| `POST /login` | публичный | Проверка пары email+пароль → сессия на 180 дней → редирект на `/`; отказ — та же форма с ошибкой, email сохранён |
| `POST /logout` | любой | Сброс сессии (включая выбор профиля) → редирект на `/login`; без сессии просто редиректит |
| `POST /profile/{member_id}` | сессия | Переключить активный профиль (член своей семьи) → 303 на `/`; чужой и несуществующий id неразличимы (404) |
| `GET /records/new` | сессия | Экран добавления записи (камера/файлы/заметка/пациент) |
| `POST /records` | сессия | Создание записи (multipart): атомарно БД+диск → 303 на `/` с тостом; доменные ошибки → re-render с текстом; чужой пациент → 404 |
| `GET /records/{id}/files/{pos}` | сессия + своя семья | Файл записи inline; чужая/несуществующая/удалённая — 404 неразличимо |
| `GET /records/{id}` | сессия + своя семья | Ветвление (поток просмотра, t₂): подтверждённая → карточка read-only (страницы вертикальным потоком, PDF и пропавший файл — плашки); неподтверждённая → экран проверки (черновик, статус текстом, meta-refresh при активном конвейере) |
| `POST /records/{id}/confirm` | сессия + своя семья | Сохранить правки полей + `confirmed_at` (один раз, ADR-012); невалидная дата → ошибка «что делать»; `return_to=card` (белый список) → 303 на карточку, иначе на ленту |
| `GET /records/{id}/edit` | сессия + своя семья | Режим правки (Э6): подтверждённая → экран проверки без AI-кодов и ретрая; неподтверждённая → 303 на `/records/{id}`; чужая/удалённая → 404 |
| `GET /records/{id}/delete` | сессия + своя семья | Страница подтверждения удаления («лист», без JS); работает и для неподтверждённых |
| `POST /records/{id}/delete` | сессия + своя семья | Мягкое удаление: `deleted_at` + `deleted_by_account_id` → 303 на `/` с тостом «Запись удалена»; повторный POST → 404 (двойной сабмит безопасен) |
| `POST /records/{id}/reparse` | сессия + своя семья | «Разобрать ещё раз»: только из `parse_failed`/зависших (>10 мин), иначе no-op |
| `GET /health` | публичный | Liveness-проба процесса: `{"status": "ok"}`, БД сознательно не проверяет |
| `GET /static/*` | публичный | Статика приложения (стили, шрифты). Файлы записей будут раздаваться НЕ отсюда |

Всё, чего нет в списке публичного (`app/middleware.py`), закрыто по умолчанию.
