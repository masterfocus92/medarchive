# DESIGN.md — правила визуальной системы

> Пара к рабочему киту `design-kit.html` — он источник истины по значениям (токены, CSS примитивов).
> Этот документ — **правила применения**: семантика, композиция, запреты. По нему собираются экраны и проводится дизайн-ревью.
> Иерархия: `design-kit.html` (значения) → `DESIGN.md` (правила) → шаблоны экранов.

---

## 1. Метафора и её смысл

Стилистика: **бумага / чернила / карандаш**. Классика, светлый фон, без изображений и иконографических излишеств.

Метафора не декоративная — она кодирует доменную модель:

| Визуальный код | Домен |
|---|---|
| **Чернила** (`--ink`) | Факты из документа: то, что разобрал AI или ввёл человек как данные |
| **Карандаш** (`--pencil`, serif italic) | Пометки человека: комментарий, заметка на полях — то, чего нет в документе |
| **Красный карандаш** (`--red-pen`) | Ошибка и удаление — правка, как в проверенной тетради |
| **Синие чернила** (`--ink-blue`) | Только фокус и ссылки |
| **Зелёные чернила** (`--green-ink`) | Только подтверждение |

**Главное правило системы:** глаз должен отличать «что в документе» от «что от себя» без чтения подписей. Если на экране факт и пометка выглядят одинаково — это дизайн-баг, а не вкусовщина.

---

## 2. Токены — единственный источник стиля

- Все значения (цвет, шрифт, размер, отступ, радиус) берутся **только из CSS-переменных** кита (`tokens.css`, извлечён из `:root` кита).
- **Сырых значений в шаблонах и стилях экранов нет.** `#1E1C19` в коде экрана — блокер на ревью; правильно — `var(--ink)`.
- Других цветов в системе не существует. Нужен новый — это изменение системы (см. §8), а не строчка в экране.
- Отступы — только шкала `--s1…--s7` (шаг 4px). Произвольные `margin: 13px` — блокер.

---

## 3. Типографика

| Шрифт | Роль | Нельзя |
|---|---|---|
| **Source Serif 4** | Заголовки, названия записей, карандашные пометки (italic) | Формы, кнопки, длинное тело |
| **Inter** | Интерфейс, тело, формы, кнопки | Заголовки записей |
| **JetBrains Mono** | Данные: даты, измерения, номера — «записано в бланк». Также служебные подписи-лейблы (uppercase, letter-spacing) | Обычный текст |

Шкала размеров — только `--t-caption…--t-h1`. Правило смысла: **дата и число всегда моноширинные** — и в выводе (`rec-date`), и во вводе (`.control.data`).

## 4. Примитивы — закрытый набор

Экраны собираются **только** из примитивов кита:

1. **Кнопки** — `btn-primary` / `btn-secondary` / `btn-quiet` / `btn-danger` (+ `btn-lg`).
2. **Поле** — `field` (label + control + helper/error), вариант `.data` для дат/чисел.
3. **Карандашная заметка** — `pencil-note` (вывод) и `textarea.pencil` (ввод).
4. **Чип** — тип записи, открытое поле.
5. **Монограмма + переключатель** — `mono-badge`, `switcher` (инициалы «имя+фамилия», без фото). Элементы `.who` — настоящие кнопки форм: переключение работает без JS.
6. **Карточка / лист** — `card`.
7. **Запись** — `record` (signature-композиция: head / meta / content / pencil-note / actions).
8. **Пустой экран** — `empty` (приглашение, не заглушка).
9. **Тост** — `toast` (подтверждение).
10. **Шаги** — `steps` (только для реальных последовательностей).
11. **Шапка приложения** — `app-head` (композиция: `switcher` слева + обособленный `btn-quiet` «Выйти» справа). Верх каждого экрана приложения; на экране входа шапка пуста.

В Jinja2 каждый примитив — макрос/partial. Один примитив = одно место в коде; копипаста разметки примитива по экранам — блокер.

## 5. Правила композиции

- **Одно первичное действие на экран.** Одна `btn-primary` в зоне видимости. Два «Сохранить» — ошибка сценария, не вёрстки.
- **Удаление всегда обособлено:** `btn-danger`, отодвинута от остальных действий (`margin-left: auto` в `rec-actions`), всегда с подтверждением.
- **Имя действия сквозное:** «Сохранить» → тост «Запись сохранена». Не «Отправить» → «Готово».
- **Ошибка не извиняется и говорит, что делать:** «Такой даты нет. Проверьте день и месяц» — а не «Упс, что-то пошло не так».
- **Ничего не блокирует сохранение:** обязательных полей в формах записи нет (кроме инварианта файл/комментарий и пациента); helper-подсказки объясняют, что AI подставит сам.
- **Статус разбора — всегда виден:** `загружено` / `разбирается` / `разобрано` / `разбор не удался` — текстом, без спиннеров-заглушек, скрывающих состояние.
- **Пустое состояние обязательно** для каждого списка: заголовок + одна фраза + первичное действие.

## 6. Layout и платформа

- Продукт — PWA, основной клиент — **телефон**. Проектировать от узкого экрана; кит держит брейкпоинт 520px.
- Контентная колонка — max-width 780px, центр; на десктопе то же самое, просто с воздухом.
- Никаких сайдбаров, табов и многоколоночных layout в POC — одна колонка, вертикальный поток.
- Тач-цели действий ≥ 44px высоты (кнопки кита это дают через паддинги — не ужимать).

---

## 7. Доступность (минимум, но обязательный)

- Фокус видим всегда: `:focus-visible` с синими чернилами — не отключать outline.
- `prefers-reduced-motion` уважается (в ките уже есть — сохранять при добавлении анимаций).
- Контраст: чернила на бумаге проходят AA; `--pencil-soft` — только для подписей, не для содержательного текста.
- Активное состояние (`aria-selected`) — семантическими атрибутами, не только классом.

---

## 8. Расширение системы

Новый примитив или токен вводится **осознанно, а не по ходу экрана**:

1. Сначала — проверка: собирается ли нужное из существующих примитивов? (Почти всегда — да.)
2. Если нет — примитив добавляется в `design-kit.html` с примером и заметкой-обоснованием.
3. Затем — сюда, в §4.
4. Только потом используется в экране.

Порядок жёсткий, потому что альтернатива — десять слегка разных карточек через месяц.

## 9. Чек-лист дизайн-ревью (для `REVIEW.md`)

- [ ] В диффе нет сырых цветов/размеров/шрифтов — только токены.
- [ ] Факты и пометки визуально разделены (чернила vs карандаш).
- [ ] Даты и измерения — моноширинные.
- [ ] Одна `btn-primary` на экран; удаление обособлено и с подтверждением.
- [ ] Тексты ошибок говорят, что делать; не извиняются.
- [ ] Есть пустое состояние и все статусы разбора.
- [ ] Разметка примитивов — через макросы, без копипасты.
- [ ] Фокус видим, reduced-motion уважается.
- [ ] Новых примитивов «по ходу» нет; если есть — они прошли §8.
